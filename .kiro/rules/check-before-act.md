# 操作前检查现状

用户经常在对话之间手动编辑、删除、移动、提交文件。

## 规则

1. 在修改任何文件之前，先确认文件的当前实际状态（用 git status、readFile 等），不要依赖之前对话的记忆或上下文传递的信息
2. 文件可能已被用户删除、移动或重命名
3. 文件内容可能已被用户手动修改
4. 之前的改动可能已被用户提交或回退
5. 如果发现现状与预期不符，先向用户确认再操作

## 工作空间说明

- 这是一个 Obsidian vault，使用 PARA + Zettelkasten 方法论
- `4-Archives/Notes/` 在 `.gitignore` 中，不要用 `git add -f` 提交里面的文件
- Wikilinks 使用 Obsidian 格式 `[[path/to/file]]`，不带 `.md` 后缀，不带别名
- 用户主要使用中文

## Git 提交规则

- 不要频繁提交 git，除非用户明确要求"提交代码"
- 每天快结束时可以自动提交一次（汇总当天改动）
- 用户也可以随时要求提交

## 日记写作规范

- 日记中的工作进展用普通列表项（`- xxx`），不要用任务项（`- [ ] xxx`）
- 已经在项目任务清单（`1-任务.md`）里的任务不要在日记里重复创建
- 日记里可以有临时任务（`- [ ] xxx`），但仅限无法归类到项目或其他任务清单的临时事项
- 工作进展列表项应加上对应的项目 tag（如 `#project/广州机场`），便于项目总览的 Dataview 自动聚合
- 子项内容写在缩进列表下，Dataview 会自动展示
