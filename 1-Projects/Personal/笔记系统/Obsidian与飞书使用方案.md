# Obsidian 与飞书使用方案

## 定位分工

| 维度 | Obsidian | 飞书 |
|------|---------|------|
| 定位 | 个人知识管理中心 | 团队协作平台 |
| 笔记 | 所有笔记的主库（PARA） | 团队共享文档（知识库） |
| 任务 | 本地视图 + 日记看板 | 权威数据源 + 团队协同 |
| 文档 | Markdown 长期存储 | 在线协作编辑 |
| 同步 | 飞书 ↔ Obsidian 双向同步（lark-cli） | — |

核心原则：**飞书负责团队协作，Obsidian 负责个人沉淀**。

## 一、目录架构

### 判断标准

| 层级 | 判断 | 放什么 |
|------|------|--------|
| Projects | 有终点吗？做完就归档 | 客户项目、产品迭代、内部改进 |
| Areas | 持续负责吗？有任务跟踪 | 按公司职能分，随业务演进 |
| Resources | 做事时查吗？定下来不变 | 技术参考、商业资料 |
| Archives | 不活跃了？冷藏 | P/A 的镜像归档 |

Areas vs Resources 的细分标准：**谁在驱动它改？**
- 随产品/业务演进会改的 → Areas（正在管）
- 决定了就不变的 → Resources（查阅用）

### 归档规则

| 场景 | 操作 | 判断标准 |
|------|------|---------|
| 项目完结 | `1-Projects/Work/X/` → `4-Archives/Projects/Work/X/` | 项目交付完成 |
| 售前没谈成 | `2-Areas/Work/业务管理/X/` → `4-Archives/Areas/Work/业务管理/X/` | 飞书文档最后更新在 2026 年之前 |
| 供应商不再合作 | `2-Areas/Work/设备管理/X/` → `4-Archives/Areas/Work/设备管理/X/` | 超过 6 个月没动 |
| 展会结束 | 展会文档移到 `4-Archives/Areas/Work/品牌宣传/` | 展会结束 |

**售前项目归档判断标准**：飞书文档最后更新日期在当年之前 → 归档。2026-04-22 通过飞书 API 验证，38 个售前项目全部在 2026 年之前更新，已归档。

频率：每季度扫一次，超过 3 个月没动的考虑归档。

### 任务格式

```markdown
- [ ] 任务标题 [](https://applink.feishu.cn/client/todo/detail?guid=xxx&suite_entity_num=txxxxx) 👤负责人 📅 2026-03-25 🛫 2026-03-20 🔺 #task/work #project/广州机场 #status/doing ^task-xxx
```

字段说明：
- `[](url)` — 飞书任务链接，通过 guid 匹配
- `👤姓名` — 负责人
- `📅` 截止日期 / `🛫` 开始日期 / `✅` 完成日期
- `🔺` 高 / `⏫` 中 / `🔼` 低优先级
- `#task/work` — 任务类型标签
- `#project/xxx` — 所属项目标签
- `#status/doing` — 状态标签（doing/testing/waiting/done）
- `^task-xxx` — block ID，用于日记中的 wikilink 引用

### 日记任务视图

每日日记通过 Tasks 插件查询自动聚合当天相关任务，按项目 → section → 状态三级分组：

```tasks
filter by function ...
group by function ... #project/xxx ...
group by function ... heading ...
group by function ... #status/xxx ...
```

### 项目任务汇报

通过 Kiro AI 对话触发，按 [[project-report-template]] 模板发送到飞书群：
1. 第一条：任务进展（已完成 + 进行中 + 总结 + 注意事项）
2. 第二条：各负责人注意事项（@ 提醒）

## 三、文档协作

### 场景一：我写的文档需要团队看

1. 在 Obsidian 中写好 Markdown
2. 用 `lark-cli docs +create` 创建飞书文档
3. 放入对应知识库节点
4. Obsidian 中标注飞书链接（frontmatter: feishu_url）

### 场景二：飞书上的协作文档

1. 在飞书上直接编辑（多人协作）
2. 定期用 `lark-cli docs +fetch` 拉取到 Obsidian
3. 本地保留 Markdown 副本作为长期存储

### 场景三：会议纪要

飞书自动生成的智能纪要处理流程：

```
飞书根目录（未处理）
  → 阅读纪要，提炼关键信息到 Obsidian 当天日记
  → 飞书端按会议内容移动到对应项目/知识库目录
  → 移动完成 = 已提炼处理
```

分类目标：
- 项目相关会议 → 对应项目知识库
- 研发部早会 → 团队管理/早会纪要
- 项目部周会 → 团队管理/周会纪要
- 技术讨论 → 产品研发/会议纪要

不拉取到 Obsidian 归档（量大且价值密度低，关键信息已提炼到日记）。

## 四、飞书迁移经验

### 迁移概况

从飞书云空间和知识库迁移到 Obsidian，共 **5341 个文件**，完整性 100%。

### 文档类型处理

| 类型 | 处理方式 |
|------|---------|
| docx（新版文档） | `docs +fetch` 直接获取 Markdown |
| doc（旧版文档） | 导出 API → docx → pandoc 转 Markdown |
| sheet（电子表格） | `sheets +export` 导出 xlsx |
| bitable（多维表格） | `base +record-list` 读取后转 Markdown |
| file（上传附件） | `drive +download` 直接下载 |
| mindnote（思维导图） | 无法 API 导出，手动下载 .mm |

### 项目分类规则

迁移中的 101 个项目按 5 个目标路径分类：

| 目标路径 | 数量 | 判断标准 |
|---------|------|---------|
| `1-Projects/Work/` | 8 | 进行中项目 |
| `4-Archives/Projects/Work/` | 35 | 已结束独立项目（文件多、有完整项目资料） |
| `4-Archives/Areas/Work/业务管理/` | 40 | 已结束售前项目（飞书更新日期在 2026 年之前） |
| `2-Areas/Work/业务管理/` | 1 | 进行中售前项目（飞书更新日期在 2026 年及之后） |
| `2-Areas/Work/设备管理/` | 1 | 归入设备管理 |

详见 [[1-Projects/Personal/笔记迁移/阶段二-飞书归档提取到PARA迁移方案]]。

## 七、日常工作流

### 早上开工

1. 打开 Obsidian 日记 → 查看今日任务（Tasks 插件自动聚合）
2. 打开飞书 → 查看消息和会议安排
3. 如果有新飞书任务 → 通过 Kiro 对话同步到 Obsidian

### 工作中

- 写技术方案/项目文档 → Obsidian
- 需要同事协作编辑 → 飞书文档
- 新任务 → 在 Obsidian 创建，Kiro 同步到飞书
- 完成任务 → 在任一端标记，同步另一端
- 临时笔记/想法 → Obsidian 日记
- 项目任务汇报 → Kiro 按模板发送到飞书群

### 下班前

1. 检查今日任务完成情况
2. 在日记中记录当天工作进展

### 周末/月末

- 周回顾：检查各项目进度，更新任务优先级
- 月回顾：归档已完结项目，整理 Resources
- 季度清理：扫 Areas 下不活跃的内容，考虑归档
