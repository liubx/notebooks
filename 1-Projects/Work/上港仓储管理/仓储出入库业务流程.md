# 仓储管理系统 — 出入库业务流程与判断机制

## 1. 业务概述

本系统是一套基于 RFID + 蓝牙定位的仓储管理应用，运行在叉车搭载的 Android 设备上。
叉车在仓库内移动时，系统自动识别货物 RFID 标签、定位叉车位置、检测附近货位，
实现入库、出库、移库操作的半自动化。

### 1.1 核心角色

| 角色 | 说明 |
|------|------|
| 货物（Trackable） | 每件货物贴有 FUWIT_RFID 标签，标签地址是唯一标识 |
| 货位（Shelf） | 仓库中的存储位置，编号格式 `仓库-排-列-层`（如 `5-A15-13-1`） |
| 叉车（Vehicle） | 搭载 Android 设备，配备读卡器、蓝牙定位、方向传感器 |
| 出库区域（Unload Area） | 仓库出口附近的地理围栏区域，出库时货物必须到达此区域 |

### 1.2 货物状态机

```
CHECKIN (9)        →     CHECKOUT (10)       →     CHECKEDOUT (11)
  待入库           入库完成     已入库（待出库）    出库完成      已出库
                  ──────→                      ──────→
                              ↻ 移库（位置变更，状态不变）
```

### 1.3 三种操作类型

| 操作 | 触发条件 | 完成条件 | 状态变更 |
|------|---------|---------|---------|
| 入库 | 货物状态为 CHECKIN | 叉车在货位附近，检测到目标库位 | CHECKIN → CHECKOUT |
| 移库 | 货物状态为 CHECKOUT 且无今日出库单 | 叉车在货位附近，检测到目标库位 | CHECKOUT → CHECKOUT（不变） |
| 出库 | 货物状态为 CHECKOUT 且有今日出库单 | 叉车不在任何货位附近，且在出库区域内 | CHECKOUT → CHECKEDOUT |

## 2. 系统架构

```
MainActivity
  └─ BackgroundService（绑定服务）
       ├─ LocationManager        蓝牙定位（叉车实时位置）
       ├─ GpsLocationManager     GPS 定位（室外/备用）
       ├─ ShelfManager           货位占用状态（WebSocket 订阅，每 300ms 更新）
       ├─ NearbyShelfManager     附近货位检测 + 放货位预测（WebSocket + 算法）
       ├─ ReaderManager          串口/蓝牙读卡器（货物 RFID 标签检测）
       ├─ RfidManager            蓝牙 RFID 读卡器（货物 RFID 标签检测）
       └─ SensorManager          方向传感器

MainFragment
  ├─ MainViewModel              任务状态管理、API 调用
  ├─ 各 EventListener           连接 Manager 事件与业务逻辑
  └─ 定时器组                    自动重连、数据刷新、日志上报
```

### 2.1 数据源与刷新频率

| 数据源 | 方式 | 频率 | 用途 |
|--------|------|------|------|
| 叉车位置 | LocationManager（蓝牙） | 实时推送 | 定位、位置上报 |
| 货位占用 | ShelfManager（WebSocket） | 每 300ms | 哪些货位有货、上次放货位置 |
| 附近货位 | NearbyShelfManager（WebSocket） | 实时推送，300ms 清理过期 | 叉车附近有哪些货位 |
| 放货预测 | NearbyShelfManager.predict() | 每 1.5s | 预测叉车要放到哪个货位 |
| RFID 标签 | ReaderManager / RfidManager | 每 500ms 判定 | 货物发现/丢失 |
| 任务列表 | ViewModel.fetchTasks() | 每 60s | 刷新待处理任务 |

## 3. 出入库完整流程

### 3.1 阶段一：空闲等待

系统启动后，叉车处于空闲状态（`task == null`），等待发现货物。

```
任务列表（fetchTasks 每 60s 刷新）
  └─ 从服务端拉取所有 CHECKIN / CHECKOUT 状态的货物
  └─ 过滤出带有 FUWIT_RFID 模块的货物
  └─ 如果是 CHECKOUT 状态，查询今日出库清单匹配 checkoutId / checkoutDate
  └─ 按优先级排序：当前任务 > 有出库单 > 状态值
```

### 3.2 阶段二：发现货物

叉车靠近货物时，读卡器检测到 RFID 标签信号。

```
ReaderManager / RfidManager
  └─ 读卡器持续扫描，每次读到标签数据（rssi, readCount）存入 TagTask 缓存
  └─ TagTask 每 500ms 执行判定：
       缓存窗口 2 秒内，满足信号强度和次数阈值的数据条数 >= 要求次数
       → 触发 onTagFound(address)
```

MainFragment 收到 onTagFound：

```
onTagFound(address)
  ├─ 当前无任务（task == null）且无正在确认的任务（checkingTask == null）
  │    └─ 在任务列表中查找 address 匹配的货物
  │         ├─ 找到 → setCheckingTask → 弹出"开始任务"对话框
  │         │         如果是重复发现同一标签：500ms 后自动确认
  │         │         如果是首次发现：1000ms 后自动确认
  │         │         （CHECKOUT 无出库单的移库任务不自动确认，需手动）
  │         └─ 未找到 → 提示"找不到对应货物"
  │
  └─ 当前有任务（task != null）
       └─ 仅将标签加入已发现列表，不触发新任务
```

#### 快速重新接任务机制

每个 listener 内部维护一个 `lastAddress`，记录上一次触发任务的标签地址。
当同一个标签短时间内丢失后又重新被检测到时（例如叉车颠簸导致货物短暂脱离读卡器范围），
系统会缩短确认等待时间，让操作员更快恢复任务：

| 场景 | 等待时间 | 对话框自动确认延迟 | 说明 |
|------|---------|------------------|------|
| 首次发现（lastAddress != address） | 1000ms | 2000ms | 正常流程，给操作员足够反应时间 |
| 重复发现（lastAddress == address） | 500ms | 100ms | 同一货物掉落后重新叉起，几乎立即恢复任务 |

典型场景：任务进行中 → 货物颠簸掉落 → onTagLost → finishTask 失败（校验不通过）→ task 清空回到空闲 → 叉车重新叉起货物 → onTagFound → lastAddress 匹配 → 500ms 后弹框 → 100ms 自动确认 → 快速恢复任务。

### 3.3 阶段三：任务进行中

startTask 成功后，系统进入任务执行状态。

**ViewModel.startTask 做了什么：**
1. 调用 `upstreamRepository.getTask(rfidAddress)` 获取货物详情
2. 获取失败则任务启动失败
3. 设置 `_task`（当前任务）、清除 `_checkingTask`
4. 更新任务列表中的 `current` 标记
5. 触发 `fetchTasks()` 刷新

**UI 变化：**
- 显示任务卡片（货物名称、原货位、货物信息、客户、入库日期、出库日期）
- 显示"完成"按钮和货位预测状态面板
- 启动日志上报定时器（每 3s 上报预测状态）

**后台持续运行：**
- NearbyShelfManager 持续预测放货位（详见第 4 节）
- 预测结果通过 `predictShelfResult` LiveData 推送到 UI
- 地图上实时显示：上次货位（灰色）、实际货位（绿色）、附近货位（橙色）、预测货位（黄色）

### 3.4 阶段四：完成任务

完成任务有两种触发方式：

**方式 A：RFID 标签丢失（自动触发）**
```
onTagLost(address) 且 address == 当前任务的 RFID 地址
  └─ 从 predictShelfResult 中获取所有预测结果（按 weight 降序）
  └─ 弹出"完成任务"库位选择对话框（2s 后自动确认推荐库位）
```

**方式 B：手动点击"完成"按钮**
```
btFinish.onClick
  └─ 同样从 predictShelfResult 获取所有预测结果
  └─ 弹出"完成任务"库位选择对话框
```

#### 完成任务对话框（showTaskFinishDialog）

根据任务类型分流到不同的对话框：

**出库（showCheckoutTaskFinishDialog）：**
简单确认对话框，不需要选择库位（出库要求 shelf == null）。
2s 后自动确认。

**入库/移库（showCheckinTaskFinishDialog）：**
单选列表对话框，列出候选库位供操作员选择：

```
候选库位列表构建（按优先级收集，去重）：
  1. predictShelf == droppingShelf 的 droppingShelf（精确命中，最高优先）
  2. 各 delay 层的 droppingShelf（weight 高的优先）
  3. 各 delay 层的 predictShelf
  4. 各 delay 层的 nearbyShelf
  推荐库位（第一个）标记 "(推荐)"

自动确认延迟：
  - 精确命中（predictShelf == droppingShelf）：2s
  - 不匹配：5s（给操作员更多时间检查和切换）
  - 无候选库位时：不自动确认，显示"未检测到附近库位"
```

### 3.5 阶段五：finishTask 判断逻辑

finishTask 根据操作类型执行不同的校验和提交流程：

#### 入库（CHECKIN）

```
1. 获取货物信息（getTask）                    → 失败则中止："未能获取货物信息"
2. 检查当前位置（_location != null）           → 失败则中止："无法获取当前位置"
3. 上报位置到货位（sendPositionUpdate）
4. 检查 shelf != null                         → 失败则中止："未检测到库位"
5. 提交入库（updateCheckinTask）               → 失败则中止："提交入库信息失败"
6. 更新状态 CHECKIN → CHECKOUT
7. 刷新任务列表
```

#### 移库（CHECKOUT 无出库单）

```
1. 获取货物信息                               → 失败则中止
2. 检查当前位置                               → 失败则中止
3. 上报位置到货位
4. 检查 shelf != null                         → 失败则中止："未检测到库位"
5. 提交移库（updateCheckinTask，同入库接口）    → 失败则中止："提交移库信息失败"
6. 更新状态 CHECKOUT → CHECKOUT（不变）
7. 刷新任务列表
```

#### 出库（CHECKOUT 有出库单）

```
1. 获取货物信息                               → 失败则中止
2. 检查当前位置                               → 失败则中止
3. 上报位置（sendPositionUpdate(task, null)，不关联货位）
4. 检查 shelf == null                         → 失败则中止："位置还在库位附近"
5. 检查 isInUnloadAreas(position)             → 失败则中止："未在出库区域"
6. 提交出库（updateCheckoutTask）              → 失败则中止："提交出库信息失败"
7. 更新状态 CHECKOUT → CHECKEDOUT
8. 刷新任务列表
```

**入库/移库 vs 出库的核心区别：**
- 入库/移库：shelf 必须不为 null（叉车必须在货位附近）
- 出库：shelf 必须为 null（叉车必须离开货位区域）且在出库地理围栏内

### 3.6 阶段六：任务暂停/中断

| 触发方式 | 说明 |
|---------|------|
| 用户手动暂停 | 点击暂停按钮 → showTaskStopDialog → 2s 自动确认 |
| 读卡器重连 | 读卡器断开重连成功时自动 stopTask |

stopTask 清除 `_task`、`_checkingTask`，更新任务列表 current 标记，回到空闲状态。

## 4. 货位预测机制（NearbyShelfManager）

### 4.1 数据采集

```
RSEngine.subscribeNearbyEvent（WebSocket，距离阈值 FORKLIFT_DISTANCE=2.0m）
  │
  ├─ onNearbyEventChange
  │    └─ 更新 nearbyShelves 列表
  │         每个 Shelf 包含：id（如 "5-A15-13"）、distance、duration、timestamp
  │         按 distance 排序
  │         已有的 shelf 更新 duration = now - 首次出现时间（上限 30s）
  │
  └─ cleanDataTimer（每 300ms）
       └─ 移除 timestamp > 5s 的过期数据
       └─ 回调 onNearbyShelfChange → UI 显示附近货位数量
```

### 4.2 货位占用数据

ShelfManager 通过 WebSocket 订阅所有货位的实时位置数据：

```
RSEngine.subscribePosition
  └─ 每 300ms 将 positions 数据映射到 shelf.level.data
       level.data != null → 该货位有货
       level.data == null → 该货位为空
  └─ 标记 last = true 的货位 → 最近一次放货的位置（lastShelf）
```

shelves 数据同步到 NearbyShelfManager.shelfLevels，用于 predict() 中判断空位。

### 4.3 预测算法（predict）

每 1.5s 执行一次，对多个时间偏移量分别预测：

```
delay = FORKLIFT_DROP_DELAY(2000ms) - locationDelay(实际定位延迟)
delays = generateDelays(delay) → 例如 [0, 1000, 1800]
每个 delay 产生一个 PredictResult，weight = 序号（delay 越大 weight 越大）
```

**单次 predict 逻辑：**

```
1. 按 delay 过滤 nearbyShelves
   （只取 timestamp 早于 delay 的数据，补偿定位延迟，还原放货时刻的真实位置）

2. 确定当前位置特征：
   - warehouse：按 duration 加权，出现时间最长的仓库号
   - row：按 duration 加权，出现时间最长的排号
   - col：当前排中最深列（最小列号），如果 duration > 300ms 则选它

3. 预测目标货位（predictShelf）：
   a. 如果 lastShelf（上次有货的位置）在附近
      （同仓库、行差 ≤ 2、列差 ≤ 3）
      → predictShelf = lastShelf 位置的空位（或其下一列空位）
      → predictType = 1（基于历史）
   b. 否则
      → predictShelf = 当前排的第一个空位
      → predictType = 0（基于当前位置）

4. 确定实际放货位（droppingShelf）：
   如果 predictShelf 在当前位置附近（行差/列差在阈值内）
      → droppingShelf = predictShelf（采纳预测）
      → 可能触发 lockShelf（锁定 3-5s 防抖动）
   否则
      → droppingShelf = 当前最近货位

5. 输出 PredictResult：
   (droppingShelf, nearbyShelf, predictShelf, lockShelf, lastShelf, delay, weight)
```

### 4.4 最佳结果选择

MainFragment 中多处使用统一的选择逻辑：

```kotlin
val sortedResults = resultMap.values.filterNotNull().sortedByDescending { it.weight }

val best = sortedResults.find { it.predictShelf?.id == it.droppingShelf?.id }  // 精确命中
    ?: sortedResults.find { it.droppingShelf != null }                         // 有稳定放货位
    ?: sortedResults.firstOrNull()                                             // weight 最大
```

优先级说明：
1. **predictShelf == droppingShelf**：预测位和实际位一致，置信度最高
2. **droppingShelf != null**：有稳定的放货位（可能被 lockShelf 锁定）
3. **weight 最大**：延迟最大的结果，最接近真实放货时刻

## 5. RFID 标签检测机制

### 5.1 检测原理

ReaderManager 和 RfidManager 持续扫描 RFID 标签，每次读到数据（rssi 信号强度、readCount 读取次数）
存入 TagTask 的缓存列表。TagTask 每 500ms 执行一次判定。

**发现判定（onTagFound）：**
在 2 秒缓存窗口内，满足信号强度和次数阈值的数据条数 >= 要求的连续次数 → 判定为发现。

**丢失判定（onTagLost）：**
已发现的标签，缓存中没有任何满足信号强度阈值的数据 → 判定为丢失，移除缓存。

### 5.2 灵敏度模式

通过 `currentRfid` + `currentEnterMode` / `currentExitMode` 控制目标标签的阈值，
通过 `defaultEnterMode` / `defaultExitMode` 控制其他所有标签的阈值。
TagTask.run 中根据 `rfid == currentRfid` 选择使用哪组阈值。

#### ReaderManager 阈值

| 参数 | mode=0（宽松） | mode=1（严格） |
|------|---------------|---------------|
| 进入 RSSI 阈值 | ≥ -50 | ≥ -40 |
| 进入 readCount 阈值 | ≥ 0 | ≥ 1 |
| 进入连续满足次数 | 2 | 3 |
| 离开 RSSI 阈值 | ≥ -50 | ≥ -60 |

#### RfidManager 阈值

| 参数 | mode=0（宽松） | mode=1（严格） |
|------|---------------|---------------|
| 进入 RSSI 阈值 | ≥ -50 | ≥ -40 |
| 进入连续满足次数 | 2 | 3 |
| 离开 RSSI 阈值 | ≥ -50 | ≥ -60 |

**mode=0 进入**：信号 -50 就够，连续 2 次即可 → 容易发现
**mode=1 进入**：信号要 -40（更强），连续 3 次 → 难发现

**mode=0 离开**：信号低于 -50 就判定丢失 → 容易丢失
**mode=1 离开**：信号要低于 -60 才判定丢失（-55 仍保持） → 难丢失

### 5.3 动态调整策略

要理解合理的设定，先看叉车操作的物理过程：

**入库/移库的物理过程：**
1. 叉车叉起货物 → RFID 标签紧贴叉车，信号很强（rssi 接近 -20~-30）
2. 叉车载着货物在仓库内移动 → 标签始终在读卡器附近，信号持续强
3. 叉车到达目标货位，放下货物 → 标签随货物留在货位上
4. 叉车开走 → 标签与读卡器距离拉大，信号逐渐减弱直到丢失
5. onTagLost 触发 → 弹出完成对话框 → finishTask 校验 shelf != null

**出库的物理过程：**
1. 叉车到达货位，叉起货物 → 标签紧贴叉车
2. 叉车载着货物离开货位区域，驶向出库区域 → 标签始终在读卡器附近
3. 叉车到达出库区域，放下货物 → 标签随货物留在出库区域
4. 叉车开走 → 信号减弱直到丢失
5. onTagLost 触发 → finishTask 校验 shelf == null 且 isInUnloadAreas

**关键洞察：**
- onTagLost 在两种场景下都意味着"叉车放下了货物并开走"
- 入库/移库时，误触发 onTagLost 的代价很高：会弹出完成对话框并可能自动提交（2s），如果此时 shelf 恰好不为 null 就会错误入库
- 出库时，误触发 onTagLost 的代价相对低：因为还有 shelf == null 和 isInUnloadAreas 两道校验
- 叉车在货位间移动时，信号可能因遮挡/反射短暂波动，这是误触发的主要来源

#### 5.3.1 分阶段设定

通过 `currentRfid` + `currentEnterMode` / `currentExitMode` 控制目标标签，
通过 `defaultEnterMode` / `defaultExitMode` 控制其他所有标签。
"目标标签"指当前任务货物的 RFID 地址（`task.trackable.tag.modules[FUWIT_RFID].address`）。

| 阶段 | currentEnterMode | currentExitMode | defaultEnterMode | defaultExitMode | 物理场景与理由 |
|------|-----------------|----------------|-----------------|----------------|--------------|
| 空闲等待 | 0 | 0 | 0 | 0 | 所有标签都需要被快速发现，尽快弹出任务确认框（currentRfid=null，全部走 default） |
| startTask 成功 | → 1 | → 1 | → 1 | 保持 0 | 目标标签严格进入+离开：防止信号波动误触发；其他标签严格进入：防止叉车经过时误扫旁边货物 |
| 入库/移库：到达目标货位 | 保持 1 | → 0 | 保持 1 | 保持 0 | 目标标签放宽离开：叉车放下货物后尽快触发完成 |
| 入库/移库：离开目标货位 | 保持 1 | → 1 | 保持 1 | 保持 0 | 目标标签恢复严格离开：叉车还在移动中，防止经过货位时误触发 |
| 出库：全程 | 保持 1 | 保持 1 | 保持 1 | 保持 0 | 目标标签全程严格：叉车载货驶向出库区域，途中不应丢失 |
| stopTask / finishTask 成功 | → 0 | → 0 | → 0 | 保持 0 | 全部重置为宽松，回到空闲（同时 currentRfid=null） |
| 读卡器重连导致 stopTask | → 0 | → 0 | → 0 | 保持 0 | 同上 |

为什么任务进行中其他标签要 enterMode=1：
- 叉车载着目标货物在货位间穿行时，会经过大量其他货物
- 如果其他标签 enterMode=0（宽松），经过时容易触发 onTagFound
- 虽然 task != null 时 onTagFound 不会触发新任务，但会：
  - 创建不必要的 TagTask 占用资源
  - 频繁更新 readerTags/rfidTags 列表造成 UI 闪烁
  - 产生大量日志噪音
- enterMode=1 要求更强信号（-40）和更多次数（3 次），叉车快速经过时通常不满足

为什么其他标签的 exitMode 始终保持 0：
- 其他标签即使被检测到（enterMode=1 也不是完全屏蔽），丢失时不会触发任何业务逻辑
- 宽松离开让它们快速从缓存中清除，减少资源占用

#### 实现方式

ReaderManager 和 RfidManager 中的属性：

```kotlin
var currentRfid: String? = null      // 目标标签地址
var currentEnterMode: Int = 0        // 目标标签的进入模式
var currentExitMode: Int = 0         // 目标标签的离开模式
var defaultEnterMode: Int = 0        // 其他标签的进入模式
var defaultExitMode: Int = 0         // 其他标签的离开模式
```

TagTask.run 中的判定逻辑：

```kotlin
val enterMode = if (rfid == currentRfid) currentEnterMode else defaultEnterMode
val exitMode = if (rfid == currentRfid) currentExitMode else defaultExitMode
if (enterMode == 1) { ... }
if (exitMode == 1) { ... }
```

调用方式（MainFragment 中）：

```
startTask 成功时：
  readerManager.currentRfid = taskRfid
  readerManager.currentEnterMode = 1
  readerManager.currentExitMode = 1
  readerManager.defaultEnterMode = 1
  (rfidManager 同理)

predictShelfResult 变化时（仅调整目标标签 exit）：
  readerManager.currentExitMode = if (readyToDrop) 0 else 1
  (rfidManager 同理)

stopTask / finishTask 成功时：
  readerManager.currentRfid = null
  readerManager.currentEnterMode = 0
  readerManager.currentExitMode = 0
  readerManager.defaultEnterMode = 0
  (rfidManager 同理)
```

#### 5.3.2 为什么入库/移库到达货位时切 exitMode=0

叉车放下货物的过程：
1. 叉车到达货位 → predictShelf == droppingShelf（预测命中）
2. 此时 exitMode 切为 0（离开阈值 -50）
3. 叉车放下货物，开始后退
4. 标签信号从 -30 逐渐降到 -45、-50
5. 信号低于 -50 → 2 秒缓存窗口内无满足条件的数据 → onTagLost
6. 弹出完成对话框，此时 shelf != null（叉车还在货位附近）→ finishTask 成功

如果 exitMode 保持 1（阈值 -60）：
- 叉车需要开得更远（信号降到 -60 以下）才能触发 onTagLost
- 此时叉车可能已经离开货位范围，nearbyShelves 变空，shelf == null
- finishTask 校验 shelf != null 失败 → "未检测到库位"
- 这就是入库/移库场景下 exitMode=0 的核心价值：让标签在叉车刚离开时就丢失，此时还在货位附近

#### 5.3.3 为什么出库不需要动态调整 exitMode

出库的完成条件是 shelf == null 且 isInUnloadAreas：
- 叉车需要完全离开货位区域，到达出库地理围栏
- 在这个过程中货物始终在叉车上，标签信号应该持续强
- exitMode=1（严格）是合理的：防止途中信号波动误触发
- 到达出库区域放下货物后，叉车开走，信号自然降到 -60 以下触发 onTagLost
- 此时 nearbyShelves 为空（不在货位附近），isInUnloadAreas 为 true → finishTask 成功

如果出库途中 exitMode=0 导致误触发 onTagLost：
- 虽然有 shelf == null 和 isInUnloadAreas 两道校验
- 但如果叉车恰好在货位区域外、出库区域外的过道上 → 两个条件都不满足 → 完成失败
- 用户体验差：弹出完成对话框但提交失败

#### 5.3.4 predictShelfResult observer 中的动态调整逻辑

```
每次 predictShelfResult 更新时（约每 1.5s）：
  如果当前有任务（currentRfid != null）：
    检查 sortedResults 中是否存在 predictShelf.id == droppingShelf.id
      ├─ 存在 → readerManager.currentExitMode = 0  // 放宽目标标签离开阈值
      └─ 不存在 → readerManager.currentExitMode = 1 // 恢复严格离开
    (rfidManager 同理)
    currentEnterMode 始终保持 1，不调整
    defaultEnterMode / defaultExitMode 不调整
```

这个调整是实时的，随着叉车移动会在 0 和 1 之间切换：
- 叉车经过一个货位 → 短暂 predictShelf == droppingShelf → exitMode=0
- 叉车继续移动 → 条件不满足 → exitMode=1
- 叉车到达最终目标货位停下 → 持续 predictShelf == droppingShelf → exitMode 保持 0
- 放下货物，开走 → onTagLost 触发

风险点：叉车经过非目标货位时短暂 exitMode=0，如果此时信号恰好波动可能误触发。
但实际上叉车经过时速度较快，500ms 的判定周期内信号不太可能持续低于 -50，
而且 predictShelf == droppingShelf 的条件本身就要求预测位和实际位一致，经过时通常不满足。

## 7. 代码调用位置

### 7.1 属性设置插入点

需要同时对 readerManager 和 rfidManager 设置（两个读卡器独立工作）。

| 位置 | 文件 | 设置内容 |
|------|------|---------|
| showTaskStartDialog 中 startTask 回调（2 处） | MainFragment | `currentRfid=taskRfid, currentEnter=1, currentExit=1, defaultEnter=1` |
| showTaskStopDialog 中 stopTask 回调（2 处） | MainFragment | `currentRfid=null, currentEnter=0, currentExit=0, defaultEnter=0` |
| showTaskFinishDialog 中 finishTask 回调（2 处） | MainFragment | 同 stopTask（无论成功失败都重置，因为 finishTask 一进入就清空 task） |
| readerManager/rfidManager isReading observer 中 stopTask 回调（2 处） | MainFragment | 同 stopTask |
| predictShelfResult observer 中 | MainFragment | 仅 `currentExitMode = if (readyToDrop) 0 else 1` |

### 7.2 注意事项

1. `currentRfid` 标识目标标签，TagTask.run 中通过 `rfid == currentRfid` 区分目标和其他标签
2. Manager.stop() / start() 不会清除 mode 属性，只清除 cacheRfids，重连后 mode 设置保留
3. `defaultExitMode` 当前策略中始终为 0，预留给后续扩展
