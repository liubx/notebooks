# 公司测试体系建设方案

## 一、背景与目标

### 1.1 建设背景

公司主营 IoT 室内定位系统，产品涵盖定位平台（Web）、移动端 App（Android）、硬件设备交互（RFID/蓝牙基站/标签）等多个层面，同时面向多个行业客户（机场、仓储、化工厂等）进行定制化交付。

当前存在的核心问题：
- **版本上线后 BUG 较多**：如 v2.7 上线后出现前后端多个 BUG，修复成本高
- **测试覆盖不全面**：容易遗漏边界场景，尤其是硬件交互和多设备协同场景
- **缺少回归测试机制**：代码改动容易引入新问题，缺乏自动化回归手段
- **没有统一的测试标准**：各项目测试方式不一致，质量参差不齐
- **现场问题排查困难**：如上港仓储现场测试中发现的标签检测异常、误入库等问题，缺乏系统化的测试预防机制

### 1.2 建设目标

- 建立标准化的测试流程：开发自测 → 功能测试 → 集成测试 → 回归测试 → 验收
- 制定测试用例编写规范和模板，覆盖各产品线
- 明确各阶段的测试责任人和验收标准
- 逐步引入自动化测试，优先覆盖 API 接口和核心业务流程
- 建立 BUG 管理和跟踪机制，与飞书任务系统打通

## 二、现状评估

| 评估维度 | 当前状况 | 存在问题 |
|---------|---------|---------|
| 开发流程 | 需求 → 开发 → 简单自测 → 部署上线 | 缺少正式测试环节，质量依赖开发自觉性 |
| 团队规模 | 小型团队（< 20 人），1 名全职测试工程师 | 测试基本靠手动，自动化能力待建设 |
| 技术栈 | 后端 Java（RESTful API）、PostgreSQL + TimescaleDB、Web 前端、Android（Kotlin）、硬件交互（RFID/蓝牙/WebSocket） | 技术栈多样，测试复杂度高 |
| 现有测试方式 | 开发人员手动自测为主，现场实地测试 | 无标准化流程，测试覆盖随机 |
| 测试工具 | 无统一测试工具 | 缺乏工具支撑 |
| 自动化程度 | 基本为零 | 所有测试依赖人工 |
| 缺陷管理 | 飞书任务清单记录 BUG | 缺少分级分类，跟踪不系统 |
| 测试环境 | 开发环境直接测试，部分项目有独立测试服务器 | 环境管理不规范，数据混乱 |
| 项目交付模式 | 多项目并行（机场、仓储、化工厂等），fork 分支做定制化开发，私有化部署 | 各项目测试标准不统一，fork 分支间回归范围不清晰 |
| 部署方式 | 基于 K8s 部署，但手动操作，每次部署约需一天 | 部署过程缺乏自动化验证，现场部署容易出问题 |
| 数据规模 | 每秒约 3000 个位置数据点，PostgreSQL + TimescaleDB | 缺乏性能基准测试，上线后才发现性能瓶颈 |

## 三、测试策略

### 3.1 测试分层体系

根据公司产品特点，建立以下测试分层：

```
┌─────────────────────────────────────────────┐
│          现场验收测试（人工）                    │  ← 客户现场
├─────────────────────────────────────────────┤
│        部署验证测试（自动化脚本）                │  ← 每次部署必跑
├─────────────────────────────────────────────┤
│        性能/压力测试（位置数据吞吐）             │  ← 新项目上线前必跑
├─────────────────────────────────────────────┤
│        E2E 测试 / 业务流程测试                  │  ← 关键流程
├─────────────────────────────────────────────┤
│          接口/API 自动化测试                    │  ← 重点投入
├─────────────────────────────────────────────┤
│        集成测试（硬件模拟 + 服务联调）            │  ← 逐步建设
├─────────────────────────────────────────────┤
│              单元测试                          │  ← 核心算法
└─────────────────────────────────────────────┘
```

### 3.2 各层测试详细说明

#### 3.2.1 单元测试

- **覆盖重点**：
  - 定位算法（坐标计算、位置预测、货位匹配）
  - RFID 标签检测判定逻辑（信号阈值、缓存窗口、灵敏度模式切换）
  - 业务状态机（货物状态流转：CHECKIN → CHECKOUT → CHECKEDOUT）
  - 数据转换和校验逻辑
  - TimescaleDB 时序数据查询逻辑（时间范围聚合、降采样、数据保留策略）
- **责任人**：开发人员
- **覆盖率目标**：核心算法模块 ≥ 80%
- **工具**：
  - 后端（Java）：JUnit 5 + Mockito + Testcontainers（用于 PostgreSQL/TimescaleDB 集成）
  - Android（Kotlin）：JUnit 5 + MockK
  - 前端（如有）：Jest / Vitest
- **执行时机**：代码提交前本地运行，CI 构建时自动执行

#### 3.2.2 接口/API 测试（优先建设）

- **覆盖重点**：
  - 定位平台核心 API（设备管理、定位对象、位置数据、地图管理）
  - 用户认证与权限（Bearer JWT / API Key 两种授权方式）
  - OEM 定制接口的差异验证
  - 数据边界和异常输入
- **责任人**：开发人员为主
- **覆盖率目标**：核心接口 100%
- **工具**：Apifox（推荐，支持团队协作 + 自动化 + Mock）
  - 利用已有的 OpenAPI 3.0 规范文件（api.yaml）直接导入
  - 按模块组织：device、position、map、management
  - 为每个 OEM 版本维护独立的测试集
- **执行时机**：每次部署到测试环境后自动执行

#### 3.2.3 集成测试

- **覆盖重点**：
  - WebSocket 实时数据推送（位置数据、货位状态、附近货位事件）
  - 硬件设备交互模拟（RFID 读卡器数据流、蓝牙定位数据流）
  - 数据库操作（货物入库/出库/移库的数据一致性）
  - 第三方系统对接（如机场排班系统数据规范）
- **责任人**：开发人员
- **工具**：
  - WebSocket 测试：自定义测试客户端 / Postman WebSocket
  - 硬件模拟：编写 Mock 数据生成器，模拟 RFID 信号流和蓝牙定位数据
- **执行时机**：CI 构建时、重大功能变更后

#### 3.2.4 E2E / 业务流程测试

- **覆盖重点**：
  - Web 平台核心流程：登录 → 设备管理 → 地图配置 → 定位监控 → 电子围栏
  - Android App 核心流程：
    - 入库流程：空闲 → 发现货物 → 开始任务 → 到达货位 → 放下货物 → 完成入库
    - 出库流程：空闲 → 发现货物 → 开始任务 → 驶向出库区域 → 放下货物 → 完成出库
    - 移库流程：空闲 → 发现货物 → 手动确认 → 到达新货位 → 完成移库
  - 异常场景：读卡器断连重连、标签信号波动、网络中断恢复
- **责任人**：开发人员 + 项目负责人
- **工具**：
  - Web：Playwright（推荐，跨浏览器支持好）
  - Android：手动测试为主，配合测试用例清单
- **执行时机**：版本发布前回归测试

#### 3.2.5 现场验收测试

- **覆盖重点**：
  - 实际硬件环境下的定位精度验证
  - 真实场景下的业务流程走通（如仓储实际叉车操作）
  - 多设备并发场景
  - 网络环境适应性（WiFi/4G 切换、弱网）
- **责任人**：项目负责人 + 现场实施人员
- **工具**：标准化的验收测试清单（Checklist）
- **执行时机**：项目交付前

#### 3.2.6 性能测试（重点建设）

公司业务特点决定了性能测试不是"按需"而是必须项。每秒 3000 个位置数据点的写入和查询，加上多项目私有化部署到不同规格的服务器，性能问题是现场部署出问题的主要原因之一。

- **覆盖重点**：
  - **位置数据写入吞吐量**：模拟 3000+ 点/秒的位置数据写入 TimescaleDB，验证不丢数据、延迟可控
  - **位置数据查询性能**：历史轨迹查询、热力图聚合、时间范围查询在大数据量下的响应时间
  - **WebSocket 长连接稳定性**：模拟 500+ 设备同时在线，持续推送位置数据，验证连接不断、内存不泄漏
  - **API 并发承载**：核心 API（设备列表、定位对象查询、围栏判定）在并发请求下的响应时间和错误率
  - **地图渲染性能**：前端同时显示 500+ 定位点时的帧率和内存占用
  - **数据库连接池压力**：高并发下 PostgreSQL 连接池是否耗尽
- **责任人**：开发人员（编写脚本）+ 测试工程师（执行和分析）
- **工具**：
  - k6（推荐，JavaScript 脚本，支持 WebSocket 和 HTTP，轻量易上手）
  - 数据库压测：pgbench + 自定义 TimescaleDB 写入脚本
  - 监控：Grafana + Prometheus（K8s 环境下通常已有）
- **执行时机**：
  - 每个新项目上线前（必须，因为客户服务器规格不同）
  - 主版本发布前
  - 数据库 schema 变更后
- **性能基准**（建议建立并持续维护）：

| 场景 | 基准指标 | 说明 |
|------|---------|------|
| 位置数据写入 | ≥ 3000 点/秒，P99 延迟 < 100ms | 核心指标，直接影响定位实时性 |
| 历史轨迹查询（单对象7天） | < 2s | 轨迹回放功能依赖 |
| 热力图聚合查询 | < 5s | 数据分析模块 |
| WebSocket 推送延迟 | P95 < 500ms | 实时地图刷新体验 |
| 核心 API 响应时间 | P95 < 300ms（50 并发） | 平台基本可用性 |
| 围栏判定延迟 | < 1s（从位置更新到告警产生） | 安全相关，不能太慢 |

#### 3.2.7 部署验证测试（新增）

这是针对你们"现场部署容易出问题"这个痛点专门设计的。每次部署到新环境（尤其是客户私有化环境），都要跑一遍部署验证。

- **覆盖重点**：
  - **K8s 服务健康检查**：所有 Pod 正常运行、readiness/liveness probe 通过
  - **数据库连通性**：PostgreSQL + TimescaleDB 扩展正常加载、连接池配置正确
  - **核心服务可用性**：定位引擎启动、WebSocket 服务可连接、API 网关响应正常
  - **数据推送链路**：MQTT/UDP/Webhook 推送通道畅通
  - **配置正确性**：项目特定配置（OEM 品牌、定制接口开关、权限配置）已正确应用
  - **时区和时间同步**：服务器时间与 NTP 同步（位置数据时间戳依赖准确时间）
  - **SSL 证书和域名**：HTTPS 证书有效、域名解析正确
  - **存储和磁盘**：TimescaleDB 数据目录空间充足、日志目录不会撑满磁盘
- **责任人**：测试工程师 + 部署人员
- **工具**：
  - 部署验证脚本（Shell/Python），自动检查上述各项
  - 冒烟测试 API 集合（Apifox 导出的精简版，只跑核心接口）
- **执行时机**：每次部署到新环境后立即执行
- **产出**：部署验证报告（通过/不通过 + 具体问题清单）

## 四、Fork 分支项目的回归测试策略

### 4.1 问题分析

公司每个定制项目都会 fork 出独立分支，这带来一个核心挑战：主线改动如何安全地合并到各项目分支？项目分支的定制改动是否破坏了通用功能？

这是"上线后 bug 多"的一个重要根源——改了 A 项目的代码，合并到 B 项目后出问题，但没人测 B 项目受不受影响。

### 4.2 分层回归策略

```
┌─────────────────────────────────────────────────────────┐
│                    通用功能回归测试集                       │
│  （所有项目共用，覆盖平台核心功能，任何分支合并后必跑）        │
├─────────────────────────────────────────────────────────┤
│  项目A定制测试集  │  项目B定制测试集  │  项目C定制测试集     │
│  （机场特有功能）  │  （仓储特有功能）  │  （化工厂特有功能）  │
└─────────────────────────────────────────────────────────┘
```

- **通用功能回归测试集**（优先建设）：
  - 覆盖功能手册中所有通用模块：登录、总览、设备管理、定位对象管理、地图管理、围栏管理、告警管理、数据分析、实时地图
  - 以 API 自动化测试为主，测试工程师维护
  - 任何分支合并代码后，先跑通用回归，通过了才能继续
  - 这是防止"改一个坏一片"的核心手段

- **项目定制测试集**：
  - 每个 fork 项目维护自己的定制功能测试用例
  - 包含该项目特有的 API 接口、业务逻辑、UI 定制
  - 项目部署前，先跑通用回归 + 项目定制测试

### 4.3 分支合并测试流程

```
主线有新功能/修复 → 合并到项目分支 → 自动触发通用回归测试 → 通过 → 跑项目定制测试 → 通过 → 可部署
                                                          → 失败 → 定位冲突，修复后重跑
```

### 4.4 实操建议

1. **先梳理各项目的定制点清单**：哪些 API 是改过的、哪些页面是定制的、哪些业务逻辑是不同的
2. **在 Apifox 中按项目建立测试集**：通用集 + 各项目定制集
3. **合并代码时必须填写影响范围**：改了哪些模块，可能影响哪些项目分支
4. **测试工程师重点关注跨分支影响**：主线改动合并到项目分支后，优先验证通用功能

## 五、基础设施搭建

### 5.1 CI/CD 流水线集成

由于已经用 K8s 部署，CI/CD 流水线应该和 K8s 打通，减少手动部署的时间（从一天缩短到可控范围）。

**目标流水线**：
```
代码提交 → 静态代码检查 → 单元测试 → 构建 Docker 镜像 → 推送镜像仓库 → 部署测试环境（K8s）→ 部署验证脚本 → API 自动化回归 → 人工验收 → 部署生产/客户环境
```

**分阶段实施**：

| 阶段 | 流水线范围 | 说明 |
|------|-----------|------|
| 第一步 | 代码提交 → 单元测试 → 构建 | 最小可行，先跑起来 |
| 第二步 | + 构建 Docker 镜像 → 推送镜像仓库 | 统一构建产物，减少"我本地能跑"问题 |
| 第三步 | + 自动部署测试环境 → 部署验证 → API 回归 | 核心价值，大幅减少手动测试工作量 |
| 第四步 | + 一键部署客户环境（带验证） | 把部署时间从一天缩短到几小时 |

**关键设计**：
- 每个 fork 项目分支都有自己的 CI 配置，但共用通用回归测试集
- 构建产物统一为 Docker 镜像 + K8s 部署清单（Helm Chart 或 Kustomize）
- 客户环境部署前，先在测试环境跑通所有自动化测试
- 工具建议：GitLab CI（如已使用 GitLab）或 GitHub Actions

### 5.2 测试环境管理

| 环境 | 用途 | 数据来源 | 管理方式 |
|-----|------|---------|---------|
| 开发环境（DEV） | 开发自测 | Mock 数据 / 独立数据库 | 开发人员自行管理 |
| 测试环境（TEST） | 功能测试、API 自动化测试 | 测试数据集 | K8s namespace 隔离，统一管理，定期重置 |
| 预发布环境（STAGING） | 模拟客户环境验证 | 脱敏的客户数据 | 按项目独立 namespace |
| 客户环境（PROD） | 私有化部署 | 真实数据 | 部署验证脚本必跑，严格变更流程 |

**K8s 环境管理建议**：
- 测试环境和预发布环境部署在公司内部 K8s 集群
- 每个 fork 项目可以有独立的 namespace，避免互相干扰
- 使用 Helm values 或 Kustomize overlay 管理不同项目的配置差异
- 测试数据库定期从模板恢复，保证测试环境干净

### 5.3 测试数据管理

- **API 测试数据**：在 Apifox 中维护标准测试数据集，覆盖正常值、边界值、异常值
- **硬件模拟数据**：编写脚本模拟 RFID 标签信号流、蓝牙定位数据流，用于非现场环境测试
- **位置数据压测数据**：编写 Java 或 k6 脚本，模拟 3000+ 点/秒的位置数据写入，用于性能基准测试
- **TimescaleDB 测试数据**：准备不同数据量级的测试数据集（1天/7天/30天/90天），验证查询性能随数据量增长的变化
- **项目定制数据**：每个 fork 项目维护独立的测试数据配置，确保定制接口的差异覆盖

## 六、流程与规范

### 6.1 标准测试流程

```
需求评审 → 开发 → 开发自测 → 提测 → 功能测试 → 回归测试 → 验收 → 上线
    │                  │          │         │           │         │
    │                  │          │         │           │         └─ 产品/客户确认
    │                  │          │         │           └─ API 自动化回归
    │                  │          │         └─ 按测试用例执行
    │                  │          └─ 填写提测单
    │                  └─ 单元测试通过 + 核心流程走通
    └─ 同步输出测试要点
```

### 6.2 测试用例管理

**管理方式**：使用飞书多维表格或 Apifox 内置用例管理（第一阶段先用飞书，降低工具切换成本）

**用例编写规范**：

| 字段 | 说明 | 示例 |
|------|------|------|
| 用例编号 | 模块-功能-序号 | API-TAG-001 |
| 用例标题 | 简明描述测试目的 | 创建标签设备-正常参数 |
| 前置条件 | 测试前需要满足的条件 | 已登录，有设备管理权限 |
| 测试步骤 | 具体操作步骤 | 1. 调用 POST /api/tags ... |
| 预期结果 | 期望的输出 | 返回 200，包含 tag_id |
| 优先级 | P0/P1/P2/P3 | P1 |
| 所属模块 | 功能模块 | 设备管理 |
| 适用项目 | 通用 / 特定 OEM | 通用 |

**用例分类**：

| 类别 | 说明 | 优先级 |
|------|------|--------|
| 冒烟测试 | 核心功能是否可用（登录、设备列表、定位数据） | 每次部署必跑 |
| 功能测试 | 各模块功能完整性验证 | 提测后执行 |
| 回归测试 | 已有功能不受新改动影响 | 上线前执行 |
| 边界测试 | 异常输入、极端场景 | 重要功能覆盖 |
| 兼容性测试 | 不同浏览器/Android 版本/设备型号 | 版本发布前 |

### 6.3 缺陷管理流程

```
发现缺陷 → 飞书任务记录（分级 + 分类 + 指派）→ 开发确认 → 修复 → 验证 → 关闭
```

**缺陷分级标准**（结合公司业务特点）：

| 等级 | 定义 | 典型场景 | 修复时限 |
|-----|------|---------|---------|
| P0-致命 | 系统不可用、数据丢失、安全漏洞 | 定位服务崩溃、货物数据丢失、越权访问 | 立即修复 |
| P1-严重 | 核心功能异常、影响业务运行 | 入库/出库流程中断、定位数据不更新、标签检测失灵 | 24 小时内 |
| P2-一般 | 次要功能异常、有替代方案 | 地图显示偏移、非核心 API 报错、UI 布局问题 | 当前迭代内 |
| P3-轻微 | 体验问题、文案错误 | 按钮文字不准确、提示信息不友好 | 排期修复 |

**缺陷分类**：

| 分类 | 说明 |
|------|------|
| 功能缺陷 | 功能不符合需求或设计 |
| 接口缺陷 | API 返回错误、参数校验缺失 |
| 硬件交互 | RFID/蓝牙/WebSocket 相关问题 |
| 性能问题 | 响应慢、内存泄漏、连接不稳定 |
| 兼容性 | 特定设备/浏览器/系统版本问题 |
| 数据问题 | 数据不一致、计算错误 |

### 6.4 准入/准出标准

**提测准入标准**：
- 开发自测通过，核心流程可走通
- 单元测试通过（如已编写）
- 提测单填写完整，包含：
  - 变更说明（改了什么）
  - 影响范围（可能影响哪些模块/项目）
  - 自测结果（自测了哪些场景）
  - 部署说明（是否需要数据库变更、配置变更等）

**上线准出标准**：
- 所有 P0/P1 缺陷已修复并验证通过
- 冒烟测试全部通过
- API 自动化回归测试通过（如已建设）
- 涉及的 fork 项目分支已验证兼容性
- 产品/项目负责人验收确认

## 十一、风险与应对

| 风险 | 影响 | 应对措施 |
|-----|------|---------|
| 开发人员时间不足 | 测试工作被压缩或跳过 | 从最小可行流程开始，逐步增加；将测试纳入工时估算 |
| 硬件环境难以模拟 | 部分场景只能现场测试 | 编写硬件模拟脚本；建立标准化现场测试清单 |
| 多项目并行精力分散 | 测试标准执行不一致 | 建立通用测试基线，各项目在此基础上扩展 |
| 自动化测试维护成本 | 接口变更导致用例失效 | 优先覆盖稳定的核心接口；利用 OpenAPI 规范自动同步 |
| 团队测试意识不足 | 流程形同虚设 | 定期复盘线上问题，用实际案例说明测试价值 |
| fork 分支过多难以全部覆盖 | 某些项目分支漏测 | 通用回归测试集是底线，定制测试集按项目活跃度优先级排序 |
| 测试工程师成为瓶颈 | 一个人忙不过来 | 明确优先级（先通用回归，再定制测试），开发承担单元测试和自测责任 |
| 客户服务器环境差异大 | 部署验证脚本不够通用 | 脚本设计为可配置，核心检查项固定，环境特定检查项可扩展 |
| 性能问题在客户环境才暴露 | 现场部署后发现性能不达标 | 建立性能基准，新项目上线前按客户服务器规格做压测 |

### C. 部署验证脚本模板

```bash
#!/bin/bash
# 部署验证脚本 - 每次部署到新环境后执行
# 用法: ./deploy-verify.sh <namespace> <api-base-url>

NAMESPACE=${1:-"default"}
API_BASE=${2:-"http://localhost:8080"}
PASS=0
FAIL=0

check() {
  local name=$1
  local result=$2
  if [ $result -eq 0 ]; then
    echo "✅ $name"
    ((PASS++))
  else
    echo "❌ $name"
    ((FAIL++))
  fi
}

echo "========== K8s 服务健康检查 =========="
# 检查所有 Pod 是否 Running
kubectl get pods -n $NAMESPACE --no-headers | grep -v "Running\|Completed" > /dev/null 2>&1
check "所有 Pod 状态正常" $((1 - $?))

# 检查 Pod 重启次数
RESTARTS=$(kubectl get pods -n $NAMESPACE -o jsonpath='{.items[*].status.containerStatuses[*].restartCount}' | tr ' ' '\n' | awk '{s+=$1}END{print s}')
[ "${RESTARTS:-0}" -lt 5 ]
check "Pod 重启次数正常（总计: ${RESTARTS:-0}）" $?

echo ""
echo "========== 数据库连通性 =========="
# 检查 PostgreSQL 连接
curl -sf "$API_BASE/actuator/health" | grep -q '"db":{"status":"UP"}'
check "PostgreSQL 数据库连接正常" $?

# 检查 TimescaleDB 扩展
curl -sf "$API_BASE/actuator/health" | grep -q "timescaledb"
check "TimescaleDB 扩展正常" $?

echo ""
echo "========== 核心服务可用性 =========="
# API 网关响应
curl -sf -o /dev/null -w "%{http_code}" "$API_BASE/api/health" | grep -q "200"
check "API 网关响应正常" $?

# WebSocket 服务
curl -sf -o /dev/null "$API_BASE/ws/health"
check "WebSocket 服务可用" $?

# 登录接口
curl -sf -X POST "$API_BASE/api/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"username":"test","password":"test"}' | grep -q "token"
check "登录接口正常" $?

echo ""
echo "========== 配置检查 =========="
# 时区
TZ=$(kubectl exec -n $NAMESPACE deploy/api-server -- date +%Z 2>/dev/null)
check "服务器时区: $TZ" 0

# 磁盘空间（数据库 PVC）
DISK_USAGE=$(kubectl exec -n $NAMESPACE deploy/postgresql -- df -h /var/lib/postgresql/data 2>/dev/null | tail -1 | awk '{print $5}' | tr -d '%')
[ "${DISK_USAGE:-100}" -lt 80 ]
check "数据库磁盘使用率: ${DISK_USAGE:-未知}%" $?

echo ""
echo "========== 验证结果 =========="
echo "通过: $PASS  失败: $FAIL"
[ $FAIL -eq 0 ] && echo "🎉 部署验证通过" || echo "⚠️ 部署验证未通过，请检查失败项"
```

      rate: 3000,           // 每秒 3000 次请求
      timeUnit: '1s',
      duration: '60s',      // 持续 60 秒
      preAllocatedVUs: 100,
      maxVUs: 200,
    },
  },
  thresholds: {

    data: {

      pos: [

        Math.random() * 100,  // y
