# 公司测试体系建设方案

## 一、背景与目标

### 1.1 建设背景

公司主营 IoT 室内定位系统，产品涵盖定位平台（Web）、移动端 App（Android）、硬件设备交互（RFID/蓝牙基站/标签）等多个层面，同时面向多个行业客户（机场、仓储、化工厂等）进行定制化交付。

当前存在的核心问题：
- **版本上线后 BUG 较多**：如 v2.7 上线后出现前后端多个 BUG，修复成本高
- **测试覆盖不全面**：容易遗漏边界场景，尤其是硬件交互和多设备协同场景
- **缺少回归测试机制**：代码改动容易引入新问题，缺乏自动化回归手段
- **没有统一的测试标准**：各项目测试方式不一致，质量参差不齐
- **现场问题排查困难**：如上港仓储现场测试中发现的标签检测异常、误入库等问题，缺乏系统化的测试预防机制

### 1.2 建设目标

- 建立标准化的测试流程：开发自测 → 功能测试 → 集成测试 → 回归测试 → 验收
- 制定测试用例编写规范和模板，覆盖各产品线
- 明确各阶段的测试责任人和验收标准
- 逐步引入自动化测试，优先覆盖 API 接口和核心业务流程
- 建立 BUG 管理和跟踪机制，与飞书任务系统打通

## 二、现状评估

| 评估维度 | 当前状况 | 存在问题 |
|---------|---------|---------|
| 开发流程 | 需求 → 开发 → 简单自测 → 部署上线 | 缺少正式测试环节，质量依赖开发自觉性 |
| 团队规模 | 小型团队（< 20 人），无专职测试人员 | 测试工作由开发兼任，投入不足 |
| 技术栈 | 后端 API（RESTful）、Web 前端、Android（Kotlin）、硬件交互（RFID/蓝牙/WebSocket） | 技术栈多样，测试复杂度高 |
| 现有测试方式 | 开发人员手动自测为主，现场实地测试 | 无标准化流程，测试覆盖随机 |
| 测试工具 | 无统一测试工具 | 缺乏工具支撑 |
| 自动化程度 | 基本为零 | 所有测试依赖人工 |
| 缺陷管理 | 飞书任务清单记录 BUG | 缺少分级分类，跟踪不系统 |
| 测试环境 | 开发环境直接测试，部分项目有独立测试服务器 | 环境管理不规范，数据混乱 |
| 项目交付模式 | 多项目并行（机场、仓储、化工厂等），含 OEM 定制 | 各项目测试标准不统一，回归范围不清晰 |

## 三、测试策略

### 3.1 测试分层体系

根据公司产品特点，建立以下测试分层：

```
┌─────────────────────────────────────────────┐
│          现场验收测试（人工）                    │  ← 客户现场
├─────────────────────────────────────────────┤
│        E2E 测试 / 业务流程测试                  │  ← 关键流程
├─────────────────────────────────────────────┤
│          接口/API 自动化测试                    │  ← 重点投入
├─────────────────────────────────────────────┤
│        集成测试（硬件模拟 + 服务联调）            │  ← 逐步建设
├─────────────────────────────────────────────┤
│              单元测试                          │  ← 核心算法
└─────────────────────────────────────────────┘
```

### 3.2 各层测试详细说明

#### 3.2.1 单元测试

- **覆盖重点**：
  - 定位算法（坐标计算、位置预测、货位匹配）
  - RFID 标签检测判定逻辑（信号阈值、缓存窗口、灵敏度模式切换）
  - 业务状态机（货物状态流转：CHECKIN → CHECKOUT → CHECKEDOUT）
  - 数据转换和校验逻辑
- **责任人**：开发人员
- **覆盖率目标**：核心算法模块 ≥ 80%
- **工具**：
  - Android（Kotlin）：JUnit 5 + MockK
  - 后端：根据后端语言选择对应框架
  - 前端（如有）：Jest / Vitest
- **执行时机**：代码提交前本地运行，CI 构建时自动执行

#### 3.2.2 接口/API 测试（优先建设）

- **覆盖重点**：
  - 定位平台核心 API（设备管理、定位对象、位置数据、地图管理）
  - 用户认证与权限（Bearer JWT / API Key 两种授权方式）
  - OEM 定制接口的差异验证
  - 数据边界和异常输入
- **责任人**：开发人员为主
- **覆盖率目标**：核心接口 100%
- **工具**：Apifox（推荐，支持团队协作 + 自动化 + Mock）
  - 利用已有的 OpenAPI 3.0 规范文件（api.yaml）直接导入
  - 按模块组织：device、position、map、management
  - 为每个 OEM 版本维护独立的测试集
- **执行时机**：每次部署到测试环境后自动执行

#### 3.2.3 集成测试

- **覆盖重点**：
  - WebSocket 实时数据推送（位置数据、货位状态、附近货位事件）
  - 硬件设备交互模拟（RFID 读卡器数据流、蓝牙定位数据流）
  - 数据库操作（货物入库/出库/移库的数据一致性）
  - 第三方系统对接（如机场排班系统数据规范）
- **责任人**：开发人员
- **工具**：
  - WebSocket 测试：自定义测试客户端 / Postman WebSocket
  - 硬件模拟：编写 Mock 数据生成器，模拟 RFID 信号流和蓝牙定位数据
- **执行时机**：CI 构建时、重大功能变更后

#### 3.2.4 E2E / 业务流程测试

- **覆盖重点**：
  - Web 平台核心流程：登录 → 设备管理 → 地图配置 → 定位监控 → 电子围栏
  - Android App 核心流程：
    - 入库流程：空闲 → 发现货物 → 开始任务 → 到达货位 → 放下货物 → 完成入库
    - 出库流程：空闲 → 发现货物 → 开始任务 → 驶向出库区域 → 放下货物 → 完成出库
    - 移库流程：空闲 → 发现货物 → 手动确认 → 到达新货位 → 完成移库
  - 异常场景：读卡器断连重连、标签信号波动、网络中断恢复
- **责任人**：开发人员 + 项目负责人
- **工具**：
  - Web：Playwright（推荐，跨浏览器支持好）
  - Android：手动测试为主，配合测试用例清单
- **执行时机**：版本发布前回归测试

#### 3.2.5 现场验收测试

- **覆盖重点**：
  - 实际硬件环境下的定位精度验证
  - 真实场景下的业务流程走通（如仓储实际叉车操作）
  - 多设备并发场景
  - 网络环境适应性（WiFi/4G 切换、弱网）
- **责任人**：项目负责人 + 现场实施人员
- **工具**：标准化的验收测试清单（Checklist）
- **执行时机**：项目交付前

#### 3.2.6 性能测试（按需）

- **覆盖重点**：
  - API 并发承载能力（多设备同时上报位置数据）
  - WebSocket 长连接稳定性（大量设备同时在线）
  - 地图渲染性能（大量定位点同时显示）
- **责任人**：开发人员
- **工具**：k6（推荐，脚本化、轻量）或 JMeter
- **执行时机**：重大版本发布前、新项目上线前

## 四、基础设施搭建

### 4.1 CI/CD 流水线集成

```
代码提交 → 静态代码检查 → 单元测试 → 构建 → 部署测试环境 → API 自动化测试 → 部署预发/客户测试环境 → 验收 → 上线
```

- 每个阶段设置质量门禁，未通过则阻断流程
- 工具建议：GitLab CI（如已使用 GitLab）或 GitHub Actions
- 第一阶段先实现：代码提交 → 单元测试 → 构建，后续逐步完善

### 4.2 测试环境管理

| 环境 | 用途 | 数据来源 | 管理方式 |
|-----|------|---------|---------|
| 开发环境（DEV） | 开发自测 | Mock 数据 / 独立数据库 | 开发人员自行管理 |
| 测试环境（TEST） | 功能测试、API 自动化测试 | 测试数据集 | 统一管理，定期重置 |
| 客户测试环境 | 项目交付前验收 | 模拟客户数据 | 按项目独立部署 |
| 生产环境（PROD） | 线上运行 | 真实数据 | 运维管理，严格变更流程 |

### 4.3 测试数据管理

- **API 测试数据**：在 Apifox 中维护标准测试数据集，覆盖正常值、边界值、异常值
- **硬件模拟数据**：编写脚本模拟 RFID 标签信号流、蓝牙定位数据流，用于非现场环境测试
- **OEM 项目数据**：每个 OEM 项目维护独立的测试数据配置，确保定制接口的差异覆盖

## 五、流程与规范

### 5.1 标准测试流程

```
需求评审 → 开发 → 开发自测 → 提测 → 功能测试 → 回归测试 → 验收 → 上线
    │                  │          │         │           │         │
    │                  │          │         │           │         └─ 产品/客户确认
    │                  │          │         │           └─ API 自动化回归
    │                  │          │         └─ 按测试用例执行
    │                  │          └─ 填写提测单
    │                  └─ 单元测试通过 + 核心流程走通
    └─ 同步输出测试要点
```

### 5.2 测试用例管理

**管理方式**：使用飞书多维表格或 Apifox 内置用例管理（第一阶段先用飞书，降低工具切换成本）

**用例编写规范**：

| 字段 | 说明 | 示例 |
|------|------|------|
| 用例编号 | 模块-功能-序号 | API-TAG-001 |
| 用例标题 | 简明描述测试目的 | 创建标签设备-正常参数 |
| 前置条件 | 测试前需要满足的条件 | 已登录，有设备管理权限 |
| 测试步骤 | 具体操作步骤 | 1. 调用 POST /api/tags ... |
| 预期结果 | 期望的输出 | 返回 200，包含 tag_id |
| 优先级 | P0/P1/P2/P3 | P1 |
| 所属模块 | 功能模块 | 设备管理 |
| 适用项目 | 通用 / 特定 OEM | 通用 |

**用例分类**：

| 类别 | 说明 | 优先级 |
|------|------|--------|
| 冒烟测试 | 核心功能是否可用（登录、设备列表、定位数据） | 每次部署必跑 |
| 功能测试 | 各模块功能完整性验证 | 提测后执行 |
| 回归测试 | 已有功能不受新改动影响 | 上线前执行 |
| 边界测试 | 异常输入、极端场景 | 重要功能覆盖 |
| 兼容性测试 | 不同浏览器/Android 版本/设备型号 | 版本发布前 |

### 5.3 缺陷管理流程

```
发现缺陷 → 飞书任务记录（分级 + 分类 + 指派）→ 开发确认 → 修复 → 验证 → 关闭
```

**缺陷分级标准**（结合公司业务特点）：

| 等级 | 定义 | 典型场景 | 修复时限 |
|-----|------|---------|---------|
| P0-致命 | 系统不可用、数据丢失、安全漏洞 | 定位服务崩溃、货物数据丢失、越权访问 | 立即修复 |
| P1-严重 | 核心功能异常、影响业务运行 | 入库/出库流程中断、定位数据不更新、标签检测失灵 | 24 小时内 |
| P2-一般 | 次要功能异常、有替代方案 | 地图显示偏移、非核心 API 报错、UI 布局问题 | 当前迭代内 |
| P3-轻微 | 体验问题、文案错误 | 按钮文字不准确、提示信息不友好 | 排期修复 |

**缺陷分类**：

| 分类 | 说明 |
|------|------|
| 功能缺陷 | 功能不符合需求或设计 |
| 接口缺陷 | API 返回错误、参数校验缺失 |
| 硬件交互 | RFID/蓝牙/WebSocket 相关问题 |
| 性能问题 | 响应慢、内存泄漏、连接不稳定 |
| 兼容性 | 特定设备/浏览器/系统版本问题 |
| 数据问题 | 数据不一致、计算错误 |

### 5.4 准入/准出标准

**提测准入标准**：
- 开发自测通过，核心流程可走通
- 单元测试通过（如已编写）
- 提测单填写完整，包含：
  - 变更说明（改了什么）
  - 影响范围（可能影响哪些模块/项目）
  - 自测结果（自测了哪些场景）
  - 部署说明（是否需要数据库变更、配置变更等）

**上线准出标准**：
- 所有 P0/P1 缺陷已修复并验证通过
- 冒烟测试全部通过
- API 自动化回归测试通过（如已建设）
- 涉及的 OEM 项目已验证兼容性
- 产品/项目负责人验收确认

## 六、各产品线测试要点

### 6.1 定位平台（Web）

| 测试类型 | 测试要点 |
|---------|---------|
| API 测试 | 设备 CRUD、定位数据查询、地图管理、用户权限、电子围栏 |
| 功能测试 | 实时地图显示、设备列表筛选、历史轨迹回放、告警通知 |
| 兼容性 | Chrome、Edge、Firefox；不同分辨率 |
| 性能 | 大量设备同时在线时的地图渲染、WebSocket 推送延迟 |
| OEM 验证 | 各 OEM 版本的品牌信息、定制接口、权限差异 |

### 6.2 仓储管理 App（Android）

| 测试类型 | 测试要点 |
|---------|---------|
| 功能测试 | 入库/出库/移库完整流程、任务状态流转、货位预测准确性 |
| 硬件交互 | RFID 标签发现/丢失判定、蓝牙定位精度、读卡器断连重连 |
| 异常场景 | 网络中断恢复、标签信号波动、多标签同时检测、叉车颠簸 |
| 灵敏度 | 各模式下的信号阈值验证（宽松/严格模式切换时机） |
| 兼容性 | 不同 Android 版本、不同型号设备、不同读卡器型号 |

### 6.3 硬件设备

| 测试类型 | 测试要点 |
|---------|---------|
| 基站部署 | 定位精度验证、覆盖范围测试、信号干扰测试 |
| 标签管理 | 标签注册/绑定、电量监控、信号强度 |
| 环境适应 | 温度、湿度、金属遮挡等对定位精度的影响 |

## 七、团队能力建设

### 7.1 组织形式

鉴于公司为小型团队（< 20 人），采用以下模式：

- **开发人员兼任测试**：每个开发人员负责自己模块的单元测试和 API 测试
- **指定测试负责人**（建议：陆东）：
  - 制定和维护测试规范、用例模板
  - 组织测试用例评审
  - 管理缺陷跟踪流程
  - 推进自动化测试建设
  - 协调各项目的测试工作
- **项目负责人**：负责各自项目的验收测试和现场测试

### 7.2 培训计划

| 阶段 | 内容 | 对象 | 形式 |
|------|------|------|------|
| 第 1 月 | 测试流程规范宣讲、缺陷管理流程 | 全员 | 团队会议 |
| 第 2 月 | Apifox 使用培训、API 测试编写 | 开发人员 | 实操培训 |
| 第 3 月 | 单元测试编写实践（结合实际项目） | 开发人员 | 结对编程 |
| 持续 | 测试经验分享、典型 BUG 复盘 | 全员 | 每月一次 |

## 八、质量度量体系

### 8.1 核心指标

| 指标 | 定义 | 初期目标 | 长期目标 |
|-----|------|---------|---------|
| 线上缺陷数 | 每个版本上线后发现的 BUG 数 | 较当前减少 30% | 较当前减少 60% |
| 缺陷逃逸率 | 线上缺陷数 / 总缺陷数 | < 20% | < 5% |
| API 测试覆盖率 | 有自动化测试的接口 / 总接口数 | 核心接口 50% | 核心接口 100% |
| 提测通过率 | 提测后首轮测试通过的比例 | > 70% | > 85% |
| 平均缺陷修复时间 | 从提交到关闭的平均时间 | P0 < 8h, P1 < 48h | P0 < 4h, P1 < 24h |
| 回归测试执行率 | 上线前执行回归测试的版本比例 | > 80% | 100% |

### 8.2 度量与复盘

- 每个版本发布后输出简要测试报告（缺陷统计、遗留问题）
- 每月在团队会议中回顾质量趋势
- 每季度评估测试体系建设进展，调整优先级

## 十、风险与应对

| 风险 | 影响 | 应对措施 |
|-----|------|---------|
| 开发人员时间不足 | 测试工作被压缩或跳过 | 从最小可行流程开始，逐步增加；将测试纳入工时估算 |
| 硬件环境难以模拟 | 部分场景只能现场测试 | 编写硬件模拟脚本；建立标准化现场测试清单 |
| 多项目并行精力分散 | 测试标准执行不一致 | 建立通用测试基线，各项目在此基础上扩展 |
| 自动化测试维护成本 | 接口变更导致用例失效 | 优先覆盖稳定的核心接口；利用 OpenAPI 规范自动同步 |
| 团队测试意识不足 | 流程形同虚设 | 定期复盘线上问题，用实际案例说明测试价值 |
