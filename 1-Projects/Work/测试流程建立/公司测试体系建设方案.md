# 公司测试体系建设方案

## 一、背景与目标

### 1.1 建设背景

公司主营室内外定位平台，典型项目模式为：对接定位厂家的定位引擎 → 接收和处理位置数据 → 定位平台后台管理（Web） → 根据客户需求做定制化开发 → 私有化部署到客户环境。面向机场、化工厂、仓储等多个行业客户，几乎每个项目都有不同程度的定制开发（API/UI/业务逻辑），以 fork 分支方式管理。少数项目（如仓储）还涉及移动端 App 和 RFID 硬件交互。

当前存在的核心问题：
- **版本上线后 BUG 较多**：如 v2.7 上线后出现前后端多个 BUG，修复成本高
- **测试覆盖不全面**：容易遗漏边界场景，尤其是定制功能与通用功能的交叉影响
- **缺少回归测试机制**：主线改动合并到项目分支后，没人验证是否破坏了已有功能
- **没有统一的测试标准**：各项目测试方式不一致，质量参差不齐
- **现场部署容易出问题**：私有化部署到客户环境后才发现配置错误、性能不达标、服务异常等问题

### 1.2 建设目标

- 建立标准化的测试流程：开发自测 → 功能测试 → 集成测试 → 回归测试 → 验收
- 制定测试用例编写规范和模板，覆盖各产品线
- 明确各阶段的测试责任人和验收标准
- 逐步引入自动化测试，优先覆盖 API 接口和核心业务流程
- 建立 BUG 管理和跟踪机制，与飞书任务系统打通

## 二、现状评估

| 评估维度 | 当前状况 | 存在问题 |
|---------|---------|---------|
| 开发流程 | 需求 → 开发 → 简单自测 → 部署上线 | 缺少正式测试环节，质量依赖开发自觉性 |
| 团队规模 | 小型团队（< 20 人），1 名全职测试工程师 | 测试基本靠手动，自动化能力待建设 |
| 技术栈 | 后端 Java（RESTful API）、PostgreSQL + TimescaleDB、Web 前端、定位引擎对接（MQTT/UDP/Webhook/HTTP） | 技术栈多样，定位数据链路长，测试复杂度高 |
| 现有测试方式 | 开发人员手动自测为主，现场实地测试 | 无标准化流程，测试覆盖随机 |
| 测试工具 | 无统一测试工具 | 缺乏工具支撑 |
| 自动化程度 | 基本为零 | 所有测试依赖人工 |
| 缺陷管理 | 飞书任务清单记录 BUG | 缺少分级分类，跟踪不系统 |
| 测试环境 | 只有开发环境和线上环境，无独立测试环境 | 开发环境不稳定，线上环境不敢随便测，客户环境部署后才发现问题 |
| 项目交付模式 | 多项目并行（机场、仓储、化工厂等），fork 分支做定制化开发，私有化部署 | 各项目测试标准不统一，fork 分支间回归范围不清晰 |
| 部署方式 | 基于 K8s 部署，但手动操作，每次部署约需一天 | 部署过程缺乏自动化验证，现场部署容易出问题 |
| 数据规模 | 每秒约 3000 个位置数据点，PostgreSQL + TimescaleDB | 缺乏性能基准测试，上线后才发现性能瓶颈 |

## 三、测试策略

### 3.1 测试分层体系

根据公司产品特点，建立以下测试分层：

```
┌─────────────────────────────────────────────┐
│          现场验收测试（人工）                    │  ← 客户现场
├─────────────────────────────────────────────┤
│        部署验证测试（自动化脚本）                │  ← 每次部署必跑
├─────────────────────────────────────────────┤
│        性能/压力测试（位置数据吞吐）             │  ← 新项目上线前必跑
├─────────────────────────────────────────────┤
│        E2E 测试 / 业务流程测试                  │  ← 关键流程
├─────────────────────────────────────────────┤
│          接口/API 自动化测试                    │  ← 重点投入
├─────────────────────────────────────────────┤
│        集成测试（定位引擎对接 + 服务联调）        │  ← 逐步建设
├─────────────────────────────────────────────┤
│              单元测试                          │  ← 核心算法
└─────────────────────────────────────────────┘
```

### 3.2 各层测试详细说明

#### 3.2.1 单元测试

- **覆盖重点**：
  - 定位算法（坐标计算、坐标转换、位置平滑/滤波）
  - 围栏判定逻辑（点在多边形内判定、多楼层围栏匹配）
  - 位置数据处理（数据清洗、异常点过滤、降采样）
  - 数据转换和校验逻辑（API 参数校验、坐标系转换）
  - TimescaleDB 时序数据查询逻辑（时间范围聚合、降采样、数据保留策略）
- **责任人**：开发人员
- **覆盖率目标**：核心算法模块 ≥ 80%
- **工具**：
  - 后端（Java）：JUnit 5 + Mockito + Testcontainers（用于 PostgreSQL/TimescaleDB 集成）
  - 前端（如有）：Jest / Vitest
- **执行时机**：代码提交前本地运行，CI 构建时自动执行

#### 3.2.2 接口/API 测试（优先建设）

- **覆盖重点**：
  - 定位平台核心 API（设备管理、定位对象、位置数据查询、地图管理、围栏管理、告警管理）
  - 用户认证与权限（登录、角色权限控制、多公司/多项目隔离）
  - 定制接口的差异验证（各 fork 项目新增或修改的接口）
  - 数据边界和异常输入
- **责任人**：测试工程师为主，开发人员配合
- **覆盖率目标**：核心接口 100%
- **工具**：Apifox（推荐，支持团队协作 + 自动化 + Mock）
  - 利用已有的 OpenAPI 3.0 规范文件（api.yaml）直接导入
  - 按模块组织：device、position、map、fence、alarm、management
  - 为每个 fork 项目维护独立的定制测试集
- **执行时机**：每次部署到测试环境后自动执行

#### 3.2.3 集成测试

- **覆盖重点**：
  - **定位引擎对接**：验证从定位厂家引擎接收位置数据的完整链路（MQTT/UDP/Webhook/HTTP 各协议）
  - **WebSocket 实时数据推送**：位置数据从接收到推送给前端的全链路延迟和正确性
  - **数据存储一致性**：位置数据写入 TimescaleDB 后查询结果的正确性、时间戳精度
  - **围栏判定联动**：位置更新 → 围栏判定 → 告警产生 → 通知推送的完整链路
  - **第三方系统对接**：各项目定制的数据对接（如机场排班系统、客户自有系统）
- **责任人**：开发人员
- **工具**：
  - WebSocket 测试：自定义测试客户端 / Postman WebSocket
  - 定位数据模拟：编写 Mock 数据生成器，模拟定位引擎按各协议推送位置数据
- **执行时机**：CI 构建时、重大功能变更后、新增定位引擎对接时

#### 3.2.4 E2E / 业务流程测试

- **覆盖重点**：
  - Web 平台核心流程（参照功能手册）：
    - 系统初始化：创建公司 → 创建用户 → 创建项目 → 设置权限
    - 项目配置：设备配置 → 创建设备 → 地图配置（建筑→楼层→地图→兴趣点）→ 创建定位对象 → 设置围栏
    - 日常使用：实时地图查看 → 定位对象监控 → 围栏告警 → 数据分析（轨迹/热力图/统计）
    - 定制功能：各项目特有的业务流程
  - 异常场景：WebSocket 断连重连、定位引擎数据中断恢复、大量设备同时上线
- **责任人**：测试工程师（手动）+ 开发人员（自动化）
- **工具**：
  - Web：Playwright（推荐，跨浏览器支持好）
  - 手动测试为主，核心流程逐步自动化
- **执行时机**：版本发布前回归测试

#### 3.2.5 现场验收测试

- **覆盖重点**：
  - 定位引擎数据接收正常（各协议通道畅通，数据不丢失）
  - 实际环境下的定位精度验证（静态精度、动态轨迹连续性、楼层切换）
  - 平台核心功能在客户网络环境下正常运行
  - 定制功能验证（客户特有的业务流程、数据对接、UI 定制）
  - 多设备并发场景（大量标签同时在线时平台响应正常）
  - 网络环境适应性（客户内网、VPN、弱网等）
- **责任人**：项目负责人 + 现场实施人员
- **工具**：标准化的验收测试清单（Checklist）
- **执行时机**：项目交付前

#### 3.2.6 性能测试（重点建设）

公司业务特点决定了性能测试不是"按需"而是必须项。每秒 3000 个位置数据点的写入和查询，加上多项目私有化部署到不同规格的服务器，性能问题是现场部署出问题的主要原因之一。

- **覆盖重点**：
  - **位置数据写入吞吐量**：模拟 3000+ 点/秒的位置数据写入 TimescaleDB，验证不丢数据、延迟可控
  - **位置数据查询性能**：历史轨迹查询、热力图聚合、时间范围查询在大数据量下的响应时间
  - **WebSocket 长连接稳定性**：模拟 500+ 设备同时在线，持续推送位置数据，验证连接不断、内存不泄漏
  - **API 并发承载**：核心 API（设备列表、定位对象查询、围栏判定）在并发请求下的响应时间和错误率
  - **地图渲染性能**：前端同时显示 500+ 定位点时的帧率和内存占用
  - **数据库连接池压力**：高并发下 PostgreSQL 连接池是否耗尽
- **责任人**：开发人员（编写脚本）+ 测试工程师（执行和分析）
- **工具**：
  - Locust（已在使用，Python 脚本，支持 HTTP/WebSocket，且可通过 Python 库直接扩展 MQTT/UDP/TCP 协议压测，一个工具覆盖所有场景）
  - 如后续需要更精细的性能指标分析，可考虑迁移到 k6（JavaScript 脚本，内置更丰富的指标统计）
  - 数据库压测：pgbench + 自定义 TimescaleDB 写入脚本
  - 监控：Grafana + Prometheus（K8s 环境下通常已有）
- **执行时机**：
  - 每个新项目上线前（必须，因为客户服务器规格不同）
  - 主版本发布前
  - 数据库 schema 变更后
- **性能基准**（建议建立并持续维护）：

| 场景 | 基准指标 | 说明 |
|------|---------|------|
| 位置数据写入 | ≥ 3000 点/秒，P99 延迟 < 100ms | 核心指标，直接影响定位实时性 |
| 历史轨迹查询（单对象7天） | < 2s | 轨迹回放功能依赖 |
| 热力图聚合查询 | < 5s | 数据分析模块 |
| WebSocket 推送延迟 | P95 < 500ms | 实时地图刷新体验 |
| 核心 API 响应时间 | P95 < 300ms（50 并发） | 平台基本可用性 |
| 围栏判定延迟 | < 1s（从位置更新到告警产生） | 安全相关，不能太慢 |

#### 3.2.7 部署验证测试（新增）

这是针对你们"现场部署容易出问题"这个痛点专门设计的。每次部署到新环境（尤其是客户私有化环境），都要跑一遍部署验证。

- **覆盖重点**：
  - **K8s 服务健康检查**：所有 Pod 正常运行、readiness/liveness probe 通过
  - **数据库连通性**：PostgreSQL + TimescaleDB 扩展正常加载、连接池配置正确
  - **核心服务可用性**：定位数据接收服务启动、WebSocket 服务可连接、API 网关响应正常
  - **数据推送链路**：定位引擎数据接收通道畅通（MQTT/UDP/Webhook/HTTP，按项目实际协议验证）
  - **配置正确性**：项目特定配置（定位引擎连接参数、定制接口开关、品牌信息、权限配置）已正确应用
  - **时区和时间同步**：服务器时间与 NTP 同步（位置数据时间戳依赖准确时间）
  - **SSL 证书和域名**：HTTPS 证书有效、域名解析正确
  - **存储和磁盘**：TimescaleDB 数据目录空间充足、日志目录不会撑满磁盘
- **责任人**：测试工程师 + 部署人员
- **工具**：
  - 部署验证脚本（Shell/Python），自动检查上述各项
  - 冒烟测试 API 集合（Apifox 导出的精简版，只跑核心接口）
- **执行时机**：每次部署到新环境后立即执行
- **产出**：部署验证报告（通过/不通过 + 具体问题清单）

## 四、Fork 分支项目的回归测试策略

### 4.1 问题分析

公司每个定制项目都会 fork 出独立分支，这带来一个核心挑战：主线改动如何安全地合并到各项目分支？项目分支的定制改动是否破坏了通用功能？

这是"上线后 bug 多"的一个重要根源——改了 A 项目的代码，合并到 B 项目后出问题，但没人测 B 项目受不受影响。

### 4.2 分层回归策略

```
┌─────────────────────────────────────────────────────────┐
│                    通用功能回归测试集                       │
│  （所有项目共用，覆盖平台核心功能，任何分支合并后必跑）        │
├─────────────────────────────────────────────────────────┤
│  项目A定制测试集  │  项目B定制测试集  │  项目C定制测试集     │
│  （机场特有功能）  │  （仓储特有功能）  │  （化工厂特有功能）  │
└─────────────────────────────────────────────────────────┘
```

- **通用功能回归测试集**（优先建设）：
  - 覆盖功能手册中所有通用模块：登录、总览、设备管理、定位对象管理、地图管理、围栏管理、告警管理、数据分析、实时地图
  - 以 API 自动化测试为主，测试工程师维护
  - 任何分支合并代码后，先跑通用回归，通过了才能继续
  - 这是防止"改一个坏一片"的核心手段

- **项目定制测试集**：
  - 每个 fork 项目维护自己的定制功能测试用例
  - 包含该项目特有的 API 接口、业务逻辑、UI 定制
  - 项目部署前，先跑通用回归 + 项目定制测试

### 4.3 分支合并测试流程

```
主线有新功能/修复 → 合并到项目分支 → 自动触发通用回归测试 → 通过 → 跑项目定制测试 → 通过 → 可部署
                                                          → 失败 → 定位冲突，修复后重跑
```

### 4.4 实操建议

1. **先梳理各项目的定制点清单**：哪些 API 是改过的、哪些页面是定制的、哪些业务逻辑是不同的
2. **在 Apifox 中按项目建立测试集**：通用集 + 各项目定制集
3. **合并代码时必须填写影响范围**：改了哪些模块，可能影响哪些项目分支
4. **测试工程师重点关注跨分支影响**：主线改动合并到项目分支后，优先验证通用功能

## 五、基础设施搭建

### 5.1 CI/CD 流水线集成

由于已经用 K8s 部署，CI/CD 流水线应该和 K8s 打通，减少手动部署的时间（从一天缩短到可控范围）。

**目标流水线**：
```
代码提交 → 静态代码检查 → 单元测试 → 构建 Docker 镜像 → 推送镜像仓库 → 部署测试环境（K8s）→ 部署验证脚本 → API 自动化回归 → 人工验收 → 部署生产/客户环境
```

**分阶段实施**：

| 阶段 | 流水线范围 | 说明 |
|------|-----------|------|
| 第一步 | 代码提交 → 单元测试 → 构建 | 最小可行，先跑起来 |
| 第二步 | + 构建 Docker 镜像 → 推送镜像仓库 | 统一构建产物，减少"我本地能跑"问题 |
| 第三步 | + 自动部署测试环境 → 部署验证 → API 回归 | 核心价值，大幅减少手动测试工作量 |
| 第四步 | + 一键部署客户环境（带验证） | 把部署时间从一天缩短到几小时 |

**关键设计**：
- 每个 fork 项目分支都有自己的 CI 配置，但共用通用回归测试集
- 构建产物统一为 Docker 镜像 + K8s 部署清单（Helm Chart 或 Kustomize）
- 客户环境部署前，先在测试环境跑通所有自动化测试
- 工具建议：GitLab CI（如已使用 GitLab）或 GitHub Actions

### 5.2 测试环境管理

**当前状况**：

| 环境 | 用途 | 说明 |
|-----|------|------|
| 开发环境（DEV） | 开发自测 | 开发人员本地或公司服务器 |
| 线上环境 | 公司自有的线上服务 | 也承担部分测试验证功能 |
| 客户私有化环境 | 各项目私有化部署 | 客户自己的服务器或云服务器 |

**问题**：没有独立的测试环境，开发自测和功能测试都在开发环境进行，API 自动化回归测试也没有稳定的执行环境。直接在线上环境测试有数据污染风险。

**建议（按优先级逐步推进）**：

1. **短期（第一阶段）**：在现有开发环境中划出一套相对稳定的配置，专门用于 API 自动化回归测试。不需要额外服务器，只需要一个独立的数据库实例和固定的测试数据集。

2. **中期（第二阶段）**：搭建一套独立的测试环境（可以是公司服务器上的一个 K8s namespace），用于：
   - API 自动化回归测试的稳定执行
   - 新版本部署后的冒烟测试
   - 主线合并到项目分支后的回归验证

3. **长期**：每个活跃的 fork 项目有独立的测试 namespace，使用 Helm values 或 Kustomize overlay 管理配置差异。

**客户私有化环境的测试策略**：
- 部署前：在公司测试环境跑通所有自动化测试
- 部署后：立即执行部署验证脚本（见 3.2.7）
- 验收前：按现场验收清单逐项检查

### 5.3 测试数据管理

- **API 测试数据**：在 Apifox 中维护标准测试数据集，覆盖正常值、边界值、异常值
- **定位数据模拟**：编写脚本模拟定位引擎按 MQTT/UDP/Webhook/HTTP 各协议推送位置数据，用于非现场环境测试
- **位置数据压测数据**：编写 Java 或 k6 脚本，模拟 3000+ 点/秒的位置数据写入，用于性能基准测试
- **TimescaleDB 测试数据**：准备不同数据量级的测试数据集（1天/7天/30天/90天），验证查询性能随数据量增长的变化
- **项目定制数据**：每个 fork 项目维护独立的测试数据配置，确保定制接口的差异覆盖

## 六、流程与规范

### 6.1 标准测试流程

```
需求评审 → 开发 → 开发自测 → 提测 → 功能测试 → 回归测试 → 验收 → 上线
    │                  │          │         │           │         │
    │                  │          │         │           │         └─ 产品/客户确认
    │                  │          │         │           └─ API 自动化回归
    │                  │          │         └─ 按测试用例执行
    │                  │          └─ 填写提测单
    │                  └─ 单元测试通过 + 核心流程走通
    └─ 同步输出测试要点
```

### 6.2 测试用例管理

**管理方式**：使用飞书多维表格或 Apifox 内置用例管理（第一阶段先用飞书，降低工具切换成本）

**用例编写规范**：

| 字段 | 说明 | 示例 |
|------|------|------|
| 用例编号 | 模块-功能-序号 | API-TAG-001 |
| 用例标题 | 简明描述测试目的 | 创建标签设备-正常参数 |
| 前置条件 | 测试前需要满足的条件 | 已登录，有设备管理权限 |
| 测试步骤 | 具体操作步骤 | 1. 调用 POST /api/tags ... |
| 预期结果 | 期望的输出 | 返回 200，包含 tag_id |
| 优先级 | P0/P1/P2/P3 | P1 |
| 所属模块 | 功能模块 | 设备管理 |
| 适用项目 | 通用 / 特定项目 | 通用 |

**用例分类**：

| 类别 | 说明 | 优先级 |
|------|------|--------|
| 冒烟测试 | 核心功能是否可用（登录、设备列表、定位数据） | 每次部署必跑 |
| 功能测试 | 各模块功能完整性验证 | 提测后执行 |
| 回归测试 | 已有功能不受新改动影响 | 上线前执行 |
| 边界测试 | 异常输入、极端场景 | 重要功能覆盖 |
| 兼容性测试 | 不同浏览器/Android 版本/设备型号 | 版本发布前 |

### 6.3 缺陷管理流程

```
发现缺陷 → 飞书任务记录（分级 + 分类 + 指派）→ 开发确认 → 修复 → 验证 → 关闭
```

**缺陷分级标准**（结合公司业务特点）：

| 等级 | 定义 | 典型场景 | 修复时限 |
|-----|------|---------|---------|
| P0-致命 | 系统不可用、数据丢失、安全漏洞 | 定位服务崩溃、位置数据丢失、越权访问 | 立即修复 |
| P1-严重 | 核心功能异常、影响业务运行 | 定位数据不更新、围栏告警失效、实时地图无法加载、定位引擎数据接收中断 | 24 小时内 |
| P2-一般 | 次要功能异常、有替代方案 | 地图显示偏移、非核心 API 报错、UI 布局问题 | 当前迭代内 |
| P3-轻微 | 体验问题、文案错误 | 按钮文字不准确、提示信息不友好 | 排期修复 |

**缺陷分类**：

| 分类 | 说明 |
|------|------|
| 功能缺陷 | 功能不符合需求或设计 |
| 接口缺陷 | API 返回错误、参数校验缺失 |
| 硬件交互 | 定位引擎对接、WebSocket 推送、数据协议相关问题 |
| 性能问题 | 响应慢、内存泄漏、连接不稳定 |
| 兼容性 | 特定设备/浏览器/系统版本问题 |
| 数据问题 | 数据不一致、计算错误 |

### 6.4 准入/准出标准

**提测准入标准**：
- 开发自测通过，核心流程可走通
- 单元测试通过（如已编写）
- 提测单填写完整，包含：
  - 变更说明（改了什么）
  - 影响范围（可能影响哪些模块/项目）
  - 自测结果（自测了哪些场景）
  - 部署说明（是否需要数据库变更、配置变更等）

**上线准出标准**：
- 所有 P0/P1 缺陷已修复并验证通过
- 冒烟测试全部通过
- API 自动化回归测试通过（如已建设）
- 涉及的 fork 项目分支已验证兼容性
- 产品/项目负责人验收确认

## 七、各产品线测试要点

### 7.1 定位平台（Web）— 核心产品

这是每个项目都涉及的核心产品，测试投入优先级最高。

| 测试类型 | 测试要点 |
|---------|---------|
| API 测试 | 设备 CRUD、定位对象管理、位置数据查询、地图管理（建筑/楼层/地图/兴趣点）、围栏管理、告警管理、用户权限、数据分析接口 |
| 功能测试 | 实时地图显示与交互、设备列表筛选、历史轨迹回放、围栏绘制与告警、数据大屏、通知管理 |
| 权限测试 | 超级管理员/公司管理员/项目管理员/项目成员各角色的菜单权限和按钮权限 |
| 兼容性 | Chrome、Edge、Firefox；不同分辨率 |
| 性能 | 大量设备同时在线时的地图渲染、WebSocket 推送延迟、数据分析页面加载速度 |
| 项目定制验证 | 各 fork 项目的品牌信息、定制接口、权限差异、UI 定制、定制业务逻辑 |

### 7.2 定位引擎对接 — 每个项目的关键环节

对接定位厂家的引擎是每个项目的核心工作之一，数据链路的正确性直接影响整个平台的可用性。

| 测试类型 | 测试要点 |
|---------|---------|
| 协议对接 | MQTT/UDP/Webhook/HTTP 各协议的数据接收正确性 |
| 数据格式 | 位置数据字段完整性（坐标、时间戳、楼层、信号强度、速度等） |
| 数据质量 | 异常数据过滤（坐标越界、时间戳异常、重复数据） |
| 吞吐量 | 每秒 3000+ 位置点的接收和处理能力 |
| 断线重连 | 定位引擎连接中断后的自动恢复 |
| 多引擎 | 同一项目对接多个定位引擎时的数据合并和冲突处理 |

### 7.3 后端服务（Java + PostgreSQL/TimescaleDB）

| 测试类型 | 测试要点 |
|---------|---------|
| 数据写入 | 位置数据批量写入 TimescaleDB 的吞吐量和延迟 |
| 数据查询 | 历史轨迹、热力图聚合、围栏进出记录等时序查询性能 |
| 数据保留 | TimescaleDB 数据压缩和保留策略是否正确执行 |
| 连接池 | 高并发下数据库连接池行为（HikariCP 配置验证） |
| 内存 | 长时间运行后 JVM 内存使用情况，是否有内存泄漏 |
| WebSocket | 大量客户端同时连接时的推送稳定性和延迟 |
| 围栏引擎 | 围栏判定实时性、多楼层围栏匹配、大量围栏时的性能 |

### 7.4 K8s 部署与运维

| 测试类型 | 测试要点 |
|---------|---------|
| 部署验证 | Pod 健康检查、服务发现、配置注入正确性 |
| 弹性 | Pod 重启后服务自动恢复、数据不丢失 |
| 资源限制 | CPU/内存 limits 设置合理，不会被 OOM Kill |
| 存储 | PVC 挂载正确、TimescaleDB 数据持久化 |
| 网络 | 服务间通信正常、Ingress 配置正确、SSL 证书有效 |
| 日志 | 日志收集正常、关键错误可追溯 |

### 7.5 特殊项目：仓储管理 App（Android）

仅上港等仓储类项目涉及，不是每个项目都需要。

| 测试类型 | 测试要点 |
|---------|---------|
| 功能测试 | 入库/出库/移库完整流程、任务状态流转、货位预测准确性 |
| 硬件交互 | RFID 标签发现/丢失判定、蓝牙定位精度、读卡器断连重连 |
| 异常场景 | 网络中断恢复、标签信号波动、多标签同时检测 |
| 兼容性 | 不同 Android 版本、不同型号设备、不同读卡器型号 |

### 7.6 硬件设备（现场验收时覆盖）

| 测试类型 | 测试要点 |
|---------|---------|
| 基站部署 | 定位精度验证、覆盖范围测试、信号干扰测试 |
| 标签管理 | 标签注册/绑定、电量监控、信号强度 |
| 环境适应 | 温度、湿度、金属遮挡等对定位精度的影响 |

## 八、团队能力建设

### 8.1 组织形式

团队不到 20 人，1 名全职测试工程师，需要最大化测试工程师的产出，同时让开发人员承担基本的测试责任。

- **测试工程师（全职，核心角色）**：
  - 维护通用功能回归测试集（API 自动化测试）
  - 维护各 fork 项目的定制测试集
  - 编写和维护部署验证脚本
  - 执行手动功能测试（新功能、复杂业务流程）
  - 管理缺陷跟踪流程
  - 协调各项目的测试工作
  - 逐步学习 k6 编写性能测试脚本
  - **不需要做的事**：不需要写单元测试（那是开发的事），不需要做现场验收（那是项目负责人的事）

- **开发人员**：
  - 编写自己模块的单元测试（Java：JUnit 5）
  - 开发自测通过后才能提测
  - 填写提测单，说明影响范围
  - 修复 bug 后自验证

- **项目负责人**：
  - 负责各自项目的现场验收测试
  - 维护项目特有的验收清单
  - 确认部署验证通过

### 8.2 测试工程师工作优先级

一个人精力有限，按以下优先级分配时间：

| 优先级 | 工作内容 | 占比 | 说明 |
|--------|---------|------|------|
| P0 | 通用 API 回归测试集建设和维护 | 30% | 这是防止"上线后 bug 多"的核心手段 |
| P1 | 新功能手动测试 | 25% | 每次提测后的功能验证 |
| P2 | 部署验证脚本编写和执行 | 15% | 解决"现场部署出问题"的痛点 |
| P3 | 各项目定制测试集维护 | 15% | fork 项目的差异化验证 |
| P4 | 缺陷管理和流程推进 | 10% | 跟踪 bug 修复、推动流程落地 |
| P5 | 性能测试（逐步学习） | 5% | 先从简单的 API 压测开始 |

### 8.3 培训计划

| 阶段 | 内容 | 对象 | 形式 |
|------|------|------|------|
| 第 1 月 | 测试流程规范宣讲、缺陷管理流程 | 全员 | 团队会议 |
| 第 1 月 | Apifox 使用培训、API 测试编写 | 测试工程师 | 实操培训（优先） |
| 第 2 月 | Apifox API 测试编写 | 开发人员 | 实操培训 |
| 第 2 月 | Locust 性能测试进阶（扩展 MQTT/UDP 协议压测） | 测试工程师 | 自学 + 实操 |
| 第 3 月 | 单元测试编写实践（JUnit 5 + Mockito） | 开发人员 | 结对编程 |
| 第 3 月 | 部署验证脚本编写 | 测试工程师 + 运维 | 实操 |
| 持续 | 测试经验分享、典型 BUG 复盘 | 全员 | 每月一次 |

### 9.1 核心指标

| 指标 | 定义 | 初期目标 | 长期目标 |

| 性能基准达标率 | 性能测试各项指标达标的比例 | 建立基准 | 100% 达标 |

## 十一、风险与应对

| 风险 | 影响 | 应对措施 |
|-----|------|---------|
| 开发人员时间不足 | 测试工作被压缩或跳过 | 从最小可行流程开始，逐步增加；将测试纳入工时估算 |
| 硬件环境难以模拟 | 定位引擎对接只能在有真实数据源时测试 | 编写定位数据模拟脚本；建立标准化现场测试清单 |
| 多项目并行精力分散 | 测试标准执行不一致 | 建立通用测试基线，各项目在此基础上扩展 |
| 自动化测试维护成本 | 接口变更导致用例失效 | 优先覆盖稳定的核心接口；利用 OpenAPI 规范自动同步 |
| 团队测试意识不足 | 流程形同虚设 | 定期复盘线上问题，用实际案例说明测试价值 |
| fork 分支过多难以全部覆盖 | 某些项目分支漏测 | 通用回归测试集是底线，定制测试集按项目活跃度优先级排序 |
| 测试工程师成为瓶颈 | 一个人忙不过来 | 明确优先级（先通用回归，再定制测试），开发承担单元测试和自测责任 |
| 客户服务器环境差异大 | 部署验证脚本不够通用 | 脚本设计为可配置，核心检查项固定，环境特定检查项可扩展 |
| 性能问题在客户环境才暴露 | 现场部署后发现性能不达标 | 建立性能基准，新项目上线前按客户服务器规格做压测 |

#!/bin/bash
# 部署验证脚本 - 每次部署到新环境后执行
# 用法: ./deploy-verify.sh <namespace> <api-base-url>

NAMESPACE=${1:-"default"}
API_BASE=${2:-"http://localhost:8080"}
PASS=0
FAIL=0

check() {
  local name=$1
  local result=$2
  if [ $result -eq 0 ]; then
    echo "✅ $name"
    ((PASS++))
  else
    echo "❌ $name"
    ((FAIL++))
  fi
}

echo "========== K8s 服务健康检查 =========="
# 检查所有 Pod 是否 Running
kubectl get pods -n $NAMESPACE --no-headers | grep -v "Running\|Completed" > /dev/null 2>&1
check "所有 Pod 状态正常" $((1 - $?))

# 检查 Pod 重启次数
RESTARTS=$(kubectl get pods -n $NAMESPACE -o jsonpath='{.items[*].status.containerStatuses[*].restartCount}' | tr ' ' '\n' | awk '{s+=$1}END{print s}')
[ "${RESTARTS:-0}" -lt 5 ]
check "Pod 重启次数正常（总计: ${RESTARTS:-0}）" $?

echo ""
echo "========== 数据库连通性 =========="
# 检查 PostgreSQL 连接
curl -sf "$API_BASE/actuator/health" | grep -q '"db":{"status":"UP"}'
check "PostgreSQL 数据库连接正常" $?

# 检查 TimescaleDB 扩展
curl -sf "$API_BASE/actuator/health" | grep -q "timescaledb"
check "TimescaleDB 扩展正常" $?

echo ""
echo "========== 核心服务可用性 =========="
# API 网关响应
curl -sf -o /dev/null -w "%{http_code}" "$API_BASE/api/health" | grep -q "200"

check "WebSocket 服务可用" $?

# Locust 脚本：模拟 UDP 定位数据推送
# 执行: locust -f udp_position_test.py --headless -u 100 -r 10 --run-time 60s

import json
import random
import socket
import time
from locust import User, task, between, events

class UdpPositionUser(User):
    """模拟定位引擎通过 UDP 推送位置数据"""
    wait_time = between(0.01, 0.05)
    
    # UDP 目标地址（定位平台数据接收服务）
    host = "127.0.0.1"
    port = 9000
    
    def on_start(self):
        self.sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    
    def on_stop(self):
        self.sock.close()
    
    @task
    def send_position(self):
        """通过 UDP 发送一条位置数据"""
        payload = json.dumps({
            "deviceName": f"tag-{random.randint(0, 9999)}",
            "name": "Pos",
            "data": {
                "mapId": random.randint(1, 10),
                "pos": [random.uniform(0, 100), random.uniform(0, 100), random.uniform(0, 5)],
                "time": int(time.time() * 1000),
            }
        }).encode("utf-8")
        
        start = time.time()
        try:
            self.sock.sendto(payload, (self.host, self.port))
            elapsed = (time.time() - start) * 1000
            events.request.fire(
                request_type="UDP",
                name="发送位置数据",
                response_time=elapsed,
                response_length=len(payload),
                exception=None,
            )
        except Exception as e:
            events.request.fire(
                request_type="UDP",
                name="发送位置数据",
                response_time=0,
                response_length=0,
                exception=e,
            )
