## 3. 接口细则

### 3.1 概述

综合定位系统主要负责机场航站楼内人员、设备等定位对象的高精度室内定位，并提供统一的、标准的、完整的位置数据服务。提供的数据包括实时位置数据、历史位置数据、定位对象管理、定位标签管理、电子围栏管理、告警数据等。

综合定位系统提供的数据将通过总线（AESB）发布，实现位置共享数据的发布，所有接入系统均需要按照总线（AESB）或生产运行主题二级交换平台（二级总线）规范进行数据的接入。

### 3.2 接口总体说明

综合定位系统通过总线的MQ和API二种方式提供数据服务。

1. 二级总线下挂的生产核心系统直接对接总线，完成API的调用；通过二级总线→总线获取MQ类型数据。
2. 总线下挂系统对接总线，完成API的调用和MQ类型数据的获取。

其中：

- **MQ方式可提供数据**：推送所有位置信息（MQ-LBS-001）、告警信息（MQ-LBS-002）
- **API方式可提供数据**：获取实时位置数据、获取历史位置数据、定位对象CRUD、定位标签管理、电子围栏查询、考勤区域判断、告警列表查询、批量绑定/解绑定位对象等

### 3.3 通信链路及技术规范

挂接系统是通过基于TCP/IP的网络环境与平台进行挂接，因此需要保证挂接系统到平台接入服务器的网络保持通畅，并能根据要求开放相应的端口。

### 3.4 数据交互目的及原则

#### 3.4.1 XML/JSON数据处理

接口提供的消息采用XML/JSON标准格式。挂接系统接口程序在解析该协议所述提供的XML/JSON消息时，须采用符合业界规范的XML/JSON解析器。否则，随着综合定位系统平台的升级和扩展，挂接系统接入程序可能会无法兼容，影响双方系统的正常数据交换。比如，综合定位系统在以后的升级中，可能会在某个服务中增加一个消息标签或者字段，如果是采用符合标准的XML/JSON解析器并按照正常的解析方法，对挂接系统接入程序不会造成影响。总之，挂接方产品的XML/JSON解析模块有责任保证其不受综合定位系统平台数据规范的进一步扩展的影响。

#### 3.4.3 XML消息基本格式

XML文档字符集编码为UTF-8，文档中的所有标签为大写。

每个XML的根标签（RootTag）为`<MSG></MSG>`。每个XML消息主要包括两个部分，一个是消息头，一个是消息主体。消息头在所有消息中相同，消息主体在不同的消息内容中有不同的定义。

当某系统收到其它系统发来的请求消息时，解析消息中的发送者，并在回复消息中向平台传送接收方代码，如果平台未收到接收方代码则平台按照配置发往需要该信息的所有系统。

根标签和消息头以及消息头内容标签描述如下：

| 标签 tag | 说明 | 值类型和宽度 | 取值范围 | 样例 | 备注 |
|----------|------|-------------|----------|------|------|
| `<MSG>` | 根标签 | N/A | N/A | N/A | N/A |
| `<META>` | 消息头标签 | N/A | N/A | N/A | N/A |
| `<SNDR>` | 消息发送者，M | S{1-8} | [0-9][a-z][A-Z] | LBS | N/A |
| `<RCVR>` | 消息接收者，O | S{1-8} | [0-9][a-z][A-Z] | ORMS | |
| `<SEQN>` | 消息序号，M | N{1-6} | 1-999999 | 1111 | 取整数，最小值为1，最大值为999999，达到最大值时循环使用。 |
| `<DDTM>` | 发送时间，M | DT{1-14}/DTF(yyyyMMddHHmmssSSS) | N/A | 20260402211034000 | N/A |
| `<TYPE>` | 消息类别，M | S{1-8} | [0-9][a-z][A-Z] | DLBS | N/A |
| `<STYP>` | 子消息类型，M | S{1-8} | [0-9][a-z][A-Z] | DPOS | N/A |
| `<SORC>` | 消息源，M | S | N/A | LBS | |
| `<MGID>` | 消息ID，M | S{32-32} | GUID | A2012122-0234-816R-E7A9-7855BA84fd5B | GUID |

#### 3.4.4 JSON数据结构格式

JSON数据结构字符集编码为UTF-8。

JSON文档中的所有字段名称均需采用驼峰规则。

JSON文档中所有定义字段在数据传输中都必须存在，即便字段内容为空。

**API接口JSON数据结构规范：**

1. API返回结果中须包含固定的参数code、from、requestId、time。
2. requestId值为请求方在Header中发送过来的requestId参数的值。
3. 接口执行成功时通过data返回具体业务数据，业务数据具体参数根据实际情况进行定义，参数统一使用小驼峰命名法进行命名。
4. 接口执行失败时通过errorMsg返回错误信息。
5. 详细参数说明如下：

| 参数名称 | 类型 | 参数含义 |
|----------|------|----------|
| code | String | 接口请求状态，执行成功：200 |
| from | String | 接口服务编码 |
| requestId | String | 请求方在请求头中所带的requestId |
| time | String | 当前系统时间值，格式：yyyy-mm-dd hh:mm:ss |
| data | Object | 执行成功时返回的具体业务数据（失败时无此参数） |
| errorMsg | String | 执行失败时返回的错误信息（成功时无此参数） |

**MQ消息JSON数据结构规范：**

1. 消息体中必须包含固定的参数：producer、from、bizId、time、data。
2. data为实际业务数据，业务数据具体参数根据实际情况进行定义，参数统一使用小驼峰命名法进行命名。
3. 详细参数说明如下：

| 参数名称 | 类型 | 参数含义 |
|----------|------|----------|
| producer | String | 消息生产者，同集成应用key |
| from | String | 接口服务编码 |
| bizId | String | 消息唯一标识，同BizId |
| time | String | 消息的发布时间，格式：yyyy-mm-dd hh:mm:ss |
| data | Object | 具体的业务数据 |
