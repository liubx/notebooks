# 问题描述

展会场景：设备通过 MQTT 上传数据，中间件通过 RabbitMQ 获取最新设备位置。消息上传频率每秒 5 条，每条最多包含 200 个设备数据（100kb）。单个中间件处理时，数据最开始几秒正常，但很快延迟指数上涨。

# 原因分析

单条消息包含 50 个设备数据，处理需要 1 秒以上。最大线程数 5，上传频率 5 条/秒，直接导致线程数拉满，后续消息排队处理，延迟越来越高。

# 已完成优化（仍有 2-3 秒延迟）

1. 消息接收层级限流，限制每秒处理一条消息，避免消息在线程池堆积
2. 起最多 8 台实例进行消息消费，满足 5 条/秒的上传频率
3. 队列消息改用 Manual Ack，避免 exception 后没有 ack 而重新发送
4. 注释了设备层级的 @FlowController（启用后导致延迟增加）

# 待优化方向

1. 消息解析和处理是主要延迟来源，单条 50 个设备数据需要 1 秒处理
2. 该延迟随每条消息包含的设备数量增加而增加
